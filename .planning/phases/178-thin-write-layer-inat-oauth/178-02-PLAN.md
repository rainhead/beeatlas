<objective>
Add the `users` table to the authoritative store (D-07/D-08): BeeAtlas mints its own internal
integer user id as the durable authorship key, with the iNat login + iNat numeric id stored as
mutable *properties* of the user row. This decouples long-lived attribution from renameable iNat
logins. Delivered as a forward-only Alembic migration owned by the write layer (177 D-03) — the
nightly pipeline never migrates it.

Purpose: WRITE-02's server-derived identity needs a place to persist the internal id that later
becomes `notes.author_id`. This is the store-side foundation; the API-side upsert access lands in 178-05.
Output: `User` model + migration `0002` + store-side tests. Store-only — no `api/` dependency.

Scope note: This plan does NOT alter `notes.author_id` (still `String`); wiring notes to the
internal user id is a Phase-179 concern. 178 only creates the users table.
</objective>
