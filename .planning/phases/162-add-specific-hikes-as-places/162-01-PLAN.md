<objective>
Create the committed, list-driven curation script `data/add_hikes_as_places.py` that
acquires each WTA hike's trail centerline from OpenStreetMap (Overpass API) — or a
hand-traced GPX fallback for the OSM gaps — buffers the centerline into a ~250 m
corridor MULTIPOLYGON in a metric CRS (UTM Zone 10N), simplifies for browser weight,
and emits `[[places]]` TOML blocks ready to append to `content/places.toml`. Add a
golden-fixture test that proves the buffer chain produces a valid MULTIPOLYGON from a
fixture LineString WITHOUT a network call, and that all 14 hike slugs follow the
immutable `-trail` convention.

This plan produces the new code artifact ONLY. It does NOT yet mutate
`content/places.toml` and does NOT hit the network — the script is RUN, the 2 OSM gaps
resolved, and the entries appended, in Plan 02 (Wave 2). Splitting this way keeps file
ownership clean (this plan owns the script + its test; Plan 02 owns `places.toml`) and
keeps the buffer-correctness logic testable offline before any geometry is published.

The defining new engineering here (absent from Phase 161, which received pre-projected
EPSG:4326 polygons) is the `ST_Transform` + `ST_Buffer` metric chain. Its single most
dangerous pitfall is omitting `always_xy=true` — without it DuckDB 1.5.3 returns
`POINT(inf inf)` SILENTLY (a correctness bug with no error). The golden-fixture test's
ST_IsValid + bbox-sanity asserts are the guard against that pitfall.

Per D-03 these are ORDINARY `[[places]]` entries — no `place_type` field, no schema
change; the script emits the same TOML shape any existing place uses, so the entire
downstream pipeline (validation, filter, sidebar) is reused unchanged.

Purpose: A reproducible, reviewable script is required because hand-authoring 14
buffered-corridor MULTIPOLYGON WKTs is infeasible and slugs are immutable after first
publish.
Output: `data/add_hikes_as_places.py` + `data/tests/test_add_hikes_as_places.py`.
</objective>
