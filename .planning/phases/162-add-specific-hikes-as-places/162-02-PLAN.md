<objective>
Run the curation script from Plan 01 to append the hike corridor entries to
`content/places.toml`, resolve the 2 OSM-gap hikes (Snoqualmie Pass to Olallie Meadow,
Geyser Valley) — first via a deeper OSM query, then via a hand-traced GPX if needed, or
formally defer with a recorded reason — ratify the simplification tolerance against the
~1 MB `places.geojson` budget, and confirm the whole existing pipeline flows green
end-to-end with the new entries: assigning hike slugs to along-trail occurrences (as
multi-place membership where a corridor overlaps a parent place) and emitting a
size-verified `public/data/places.geojson`.

This is the content-addition plan. The downstream pipeline (validate → load → dbt-build →
generate-sqlite → export → maps → frontend) is REUSED UNCHANGED — no code edits beyond
appending TOML blocks (and possibly committing GPX fixtures). A hike corridor overlapping
a WDFW area or national forest must LOAD cleanly as multi-place membership (Phase 160
removed the `ST_Overlaps` rejection); do NOT clip, skip, triage, or hard-fail on overlaps.

The 2 OSM-gap hikes are the one place this plan can pause for a human: if neither a deeper
OSM query nor an existing committed GPX yields usable geometry, hand-tracing a GPX is a
manual step. That is gated by a `checkpoint:human-action` (Task 1b) — the gap hikes are
TRACKED, never silently dropped. The POC succeeds if the resolvable subset loads cleanly
and any remaining gap is explicitly resolved or deferred.

Purpose: Make bee occurrences collected along these trails filterable and render their
corridor boundaries on the map, within the browser weight budget.
Output: updated `content/places.toml` (+ up to 14 entries); optional committed GPX
fixtures for the gap hikes; a regenerated, size-verified `public/data/places.geojson`.
</objective>
