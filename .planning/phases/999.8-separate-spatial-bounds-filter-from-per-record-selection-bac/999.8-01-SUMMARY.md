# Phase 999.8 Plan 01: FilterState.bounds + buildFilterSQL bounds clause Summary

**One-liner:** Added `FilterState.bounds` field and routed it through `isFilterActive` + `buildFilterSQL` so map/list/table queries inherit the spatial bounding box as "just another filter."

## What Was Built

**Task 1** (TDD RED→GREEN): Added `bounds: { west: number; south: number; east: number; north: number } | null` to the `FilterState` interface in `src/filter.ts`, after `selectedPlace`. Added `|| f.bounds !== null` as the final clause in `isFilterActive`. Updated `emptyFilter()` helper in `filter.test.ts` to include `bounds: null`, added two new `isFilterActive` tests (bounds set → true, emptyFilter → false), and fixed the inline `FilterState` literal in the combined-filters test.

**Task 2** (TDD RED→GREEN): Added a bounds clause branch inside `buildFilterSQL` (after the elevation block) that pushes `lat BETWEEN ${south} AND ${north} AND lon BETWEEN ${west} AND ${east}` onto `occurrenceClauses` when `f.bounds !== null`. Removed the `selectionBounds` parameter from `queryVisibleGeoJSON` (signature now takes only `f: FilterState`; guard simplified from `!isFilterActive(f) && selectionBounds === null` to `!isFilterActive(f)`; inline `boundsClause` local deleted). Removed the `selectionBounds` parameter from `queryListPage` (inline `boundsClause` local and `${boundsClause}` from `fullWhere` deleted). Added three new `buildFilterSQL` tests: bounds set emits lat/lon BETWEEN, emptyFilter emits neither, bounds+taxonId AND-compose.

`queryTablePage` and `queryOccurrencesByBounds` are intentionally untouched — `queryTablePage` now inherits bounds automatically via `buildFilterSQL(f)` (the latent table-view gap is fixed for free); `queryOccurrencesByBounds` takes an explicit `bounds` arg for the shift-drag sidebar fetch and must not double-filter.
