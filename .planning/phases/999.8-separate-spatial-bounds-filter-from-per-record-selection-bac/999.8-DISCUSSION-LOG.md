# Phase 999.8: Separate spatial-bounds FILTER from per-record SELECTION - Discussion Log

**Date:** 2026-06-21
**Phase:** 999.8-separate-spatial-bounds-filter-from-per-record-selection-bac
**Areas discussed:** State model, URL param & migration, Pane behavior, Coexistence, Empty-click clear, Bounds clear affordance

---

## State model

| Option | Description | Selected |
|--------|-------------|----------|
| Fold into FilterState | Bounds becomes a field on FilterState; flows through existing filter plumbing. Biggest blast radius, most conceptually pure. | ✓ |
| Dedicated _boundsFilter field | Separate reactive field passed alongside _filterState. Smaller change, bounds stays a special case. | |

**User's choice:** Fold into FilterState
**Notes:** Chosen for conceptual purity — "bounds is just another filter." (D-01)

---

## URL param & legacy migration

| Option | Description | Selected |
|--------|-------------|----------|
| New param, read-old/write-new | New `bbox=` param; parseParams still reads legacy `sel=` and maps it in; buildParams only writes new. `sel=` retired for bounds. | ✓ |
| Keep sel= for bounds | Leave wire format as `sel=`, rename only internal state. Zero migration risk. | |
| New param, dual-read forever | New param for writes, keep parsing `sel=` indefinitely as permanent alias. | |

**User's choice:** New param, read-old/write-new
**Notes:** Param locked as `bbox=` (Claude discretion on name, confirmed via specifics). (D-02, D-03)

---

## Pane behavior

| Option | Description | Selected |
|--------|-------------|----------|
| Leave pane as-is | Bounds change does not touch _paneState; behaves like every other filter. Roadmap intent. | ✓ |
| Keep forcing list open | Preserve current force-open of list pane. Contradicts filter model. | |

**User's choice:** Leave pane as-is
**Notes:** Remove the `_paneState='list'` force in the bounds apply path. (D-04)

---

## Filter / selection coexistence

| Option | Description | Selected |
|--------|-------------|----------|
| Allow coexistence (AND) | Bounds filter can be active while records are selected; they compose. More state-combination cases. | ✓ |
| Keep mutually exclusive | Applying one clears the other, via cleaner separated code. Less risk. | |

**User's choice:** Allow coexistence (AND)
**Notes:** Logical end state of independent filter vs selection concepts. (D-05)

---

## Empty-map-click clear semantics

| Option | Description | Selected |
|--------|-------------|----------|
| Selection only | Empty-map click clears per-record selection but leaves the bounds filter active. | ✓ |
| Selection + bounds | Empty-map click clears both (closest to today). | |

**User's choice:** Selection only
**Notes:** Consistent with bounds being a filter dismissed via filter affordances, not a map click. (D-06)

---

## Bounds clear affordance

| Option | Description | Selected |
|--------|-------------|----------|
| Via the 'where' input clear | Cleared through the existing 'where' input affordance (153 mechanism). No new UI. | ✓ (primary) |
| Also via a global filter reset | Fold bounds into a 'clear all filters' reset too. | ✓ (requested) → deferred |

**User's choice:** Both — primary clear via the 'where' input; also fold into a global filter reset
**Notes:** Verified no global "clear all filters" affordance exists in the codebase today. Primary clear (where input) is in scope (D-07); global-reset participation is deferred until such an affordance exists (D-08). Folding bounds into FilterState makes the future global reset trivial.
