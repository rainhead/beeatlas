<objective>
Make `FilterState` carry the spatial bounding box as a first-class filter field, and route it through the existing filter SQL plumbing so the map, list, AND table views all respect it automatically (D-01). This is the type-level foundation: nothing in `bee-atlas.ts` changes yet ŌĆö only `filter.ts` and its tests.

Purpose: Today the bounding box rides a separate `selectionBounds` argument threaded into `queryVisibleGeoJSON`/`queryListPage`, and the table view (`queryTablePage`) never received it at all. Folding bounds into `FilterState` + `buildFilterSQL` makes bounds "just another filter," fixes the latent table-view gap, and lets `isFilterActive`/`buildParams` cover bounds with no special-casing.

Output: `FilterState.bounds` field, `isFilterActive` bounds clause, `buildFilterSQL` bounds clause, `queryVisibleGeoJSON`/`queryListPage` signatures with the `selectionBounds` parameter removed, and updated `filter.test.ts`.
</objective>
