<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server `html` → island DOM | the read endpoint's `html` field is injected via unsafeHTML; it must be trusted server-sanitized output |
| island → write API | all mutations go to require_author-guarded endpoints; the browser never enforces authz itself |
| whoami → UI affordances | isAuthor gates the editor UI only; the server re-checks the allowlist on every write (client gating is cosmetic) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-179-XSS | Tampering | unsafeHTML(note.html) | mitigate | only the server-sanitized `html` field is injected (nh3 allowlist, 179-01); the island never parses markdown or sanitizes client-side (D-04); `body_md` is only ever put into a textarea, never rendered as HTML |
| T-179-AUTHZ | Elevation of Privilege | client edit/delete affordances | accept→mitigate | isAuthor + can_edit only hide/show UI; authorization is enforced server-side by require_author + ownership (179-02) — a forged client cannot bypass it; a 403 mid-session drops the controls |
| T-179-DIVERGE | Information (expected) | live vs baked byline/list | accept | live @login vs baked full display_name, and live-note-not-yet-in-notes.json, are expected up to one nightly cycle (RESEARCH Pitfall 4) — never surfaced as an error |
</threat_model>
