<objective>
Harvest published notes into a build-time `notes.json` (D-09/D-10/D-13) — an exact
structural mirror of the shipped `species_hosts.js` bake — and wire it into the
nightly pipeline as a read-only STEP after `collectors-events-export`, declared
`authoritative` in the Phase-176 artifact contract. Add the absence-tolerant
`_data/notes.js` loader so the static read path renders notes with no runtime API
call (NOTES-03: static + offline-safe).

Purpose: This is the harvest-dominant read path. The store is the source of truth;
notes.json is disposable/re-harvestable and never committed to git — it ships via
S3 + manifest.json + deploy.yml fetch through the derived-vs-authoritative contract.
Output: `data/notes_harvest.py`, a run.py STEP, `[artifacts.notes]`, `_data/notes.js`,
and their tests (plus a narrowing of the run.py-isolation test — see Task 2).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| store → public notes.json | only approved notes may cross into the public bake; pending/removed must be excluded |
| nightly pipeline → authoritative store | run.py must read the store read-only, never migrate or write it (D-03/STORE-02) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-179-LEAK | Information Disclosure | notes_harvest query | mitigate | harvest query filters `status == 'approved'`; test asserts pending/removed never appear in notes.json (mirrors MOD-02 harvest exclusion) |
| T-179-MIG | Tampering | run.py touching the store | mitigate | run.py references only `notes_harvest`; harvest opens read-only WAL via make_engine; `test_run_py_never_migrates` keeps banning notes_store/alembic/notes.db/NOTES_DB_PATH + migrate-named steps |
| T-179-LOCK | Denial of Service | concurrent harvest read vs API write | mitigate | harvest reuses notes_store.db.make_engine (WAL + synchronous=NORMAL), never a raw sqlite3 connection (RESEARCH Pitfall 5) |
| T-179-COMMIT | Tampering | committing notes.json | mitigate | public/data/ stays gitignored; ships via S3+manifest+deploy.yml; a clean git status is explicitly NOT a PASS (memory feedback_no_committed_data_artifacts) |
</threat_model>
