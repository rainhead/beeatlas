# Phase 179 — UI Design Contract

> Visual and interaction contract for the species-page natural-history notes
> section: the baked read-only list (everyone, always, offline-safe) and the
> progressive-enhancement authoring island (allowlisted authors only).
> Generated by gsd-ui-researcher. Run unattended (`--auto`) — no open design
> questions were left to the user; every discretionary call below is recorded
> as an explicit assumption for the checker/planner to confirm.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — this is a Lit + Eleventy/Nunjucks codebase, not React. shadcn is a React-ecosystem tool and does not apply here regardless of Vite's presence (Vite is used as the bundler for Lit entries, not a React app). No `components.json`, confirmed. |
| Preset | not applicable |
| Component library | none — hand-rolled Lit `LitElement` components (`src/bee-*.ts`) + plain Nunjucks/CSS |
| Icon library | none — inline hand-drawn SVG (Heroicons-style 24×24 stroke icons, see `src/bee-header.ts`); this phase adds no new icons (notes UI is text-label buttons only, matching `.auth-btn`/`.cache-popover__update-btn` precedent) |
| Font | system-ui, Avenir, Helvetica, Arial, sans-serif (`src/index.css` `:root`) |

This is an additive feature on an established Eleventy/Nunjucks + Lit site. All tokens
come from `src/index.css` and `src/styles/taxon-pages.css`. Do not introduce a new type
family, icon set, component library, or a modal/dialog pattern — D-01 explicitly
forbids a modal for the editor.

**Naming correction:** RESEARCH.md's code sketch names the island `notes-island.ts` /
`<notes-island>`. Per `CLAUDE.md`'s Lit naming convention (`bee-*`) and this UI-SPEC's
own instructions, the custom element is named **`<bee-notes>`** (file
`src/bee-notes.ts`). Use this name in the plan; it is a design-contract-level naming
fix, not merely a file-rename detail, because CSS selectors and test IDs key off it.

## Spacing Scale

Reuses the existing site scale (`rem`, all multiples of 4px). No new spacing values.

| Token | rem | px | Usage in this phase |
|-------|-----|----|-----|
| xs | 0.25rem | 4px | Gap between byline and timestamp; note-list item internal row gaps |
| sm | 0.5rem | 8px | Gap between "Add note"/"Cancel"/"Save" buttons; gap between note-meta elements |
| md | 1rem | 16px | Space between stacked `.note` articles; editor top margin |
| lg | 1.5rem | 24px | Notes section top margin (mirrors `.collected-from`/`.traits` block spacing of `0.5rem 0 0.75rem` — see below, this section is heavier so gets more breathing room) |

Exceptions:
- `.notes-section` margin: `1rem 0 1.5rem` (16px top / 24px bottom) — larger than
  `.traits`/`.collected-from`'s `0.5rem 0 0.75rem` because notes are prose blocks, not
  compact metadata; still on-grid (16px, 24px).
- `.note + .note` gap: `1rem` (16px) — enough separation between distinct authored
  entries without a card/box treatment (see Color section — no background boxes).
- `.note-editor` internal padding: `0.75rem` (12px) — matches `.traits-dl` column-gap
  precedent for a "slightly tighter than md" content pad.

| Role | Size | Weight | Line Height | Usage |

| Body (note prose) | 1rem (16px) | 400 | 1.5 | `.note-body` — the rendered restricted-markdown content. `.note-body` typographic rules (restricted-markdown output — `p`, `em`, `strong`, `a`,

## Layout Contract

### Placement in `_pages/species-detail.njk`

Insert the notes section after the existing `.taxon-action` links (it is new
authoritative content, distinct in kind from the derived metadata/traits/hosts
blocks above it — placing it last keeps the established derived-data hierarchy
intact and matches the "differentiator" framing of NOTES-04):

```
<a class="taxon-action">View records →</a>        ← conditional (existing)
<a class="taxon-action">View on iNaturalist →</a> ← conditional (existing)
[NEW] <section class="notes-section" id="notes">   ← baked, conditional on notes existing
  ...
</section>
[NEW] <bee-notes id="notes-el"></bee-notes>         ← always present, mount point + inline script
<script>...</script>                                ← hands bee-notes canonicalName + bakedNotes
```

### Baked section (Nunjucks, always the source of truth)

Exact structural mirror of the `.collected-from` conditional-render pattern:

```nunjucks
{%- set notesForSpecies = notes[sp.canonical_name] -%}
{%- if notesForSpecies and notesForSpecies.length > 0 -%}
<section class="notes-section" id="notes">
  <h2 class="notes-heading">Natural history notes</h2>
  <div class="note-list">
    {%- for note in notesForSpecies -%}
    <article class="note" data-note-id="{{ note.id }}">
      <div class="note-body">{{ note.html | safe }}</div>
      <footer class="note-meta">
        {%- if note.byline.collector_url -%}
        <a class="note-byline" href="{{ note.byline.collector_url }}">{{ note.byline.display_name }}</a>
        {%- else -%}
        <span class="note-byline">{{ note.byline.display_name }}</span>
        {%- endif -%}
        <time class="note-timestamp" datetime="{{ note.created }}">{{ note.created | formatDate }}</time>
        {%- if note.updated !== note.created -%}
        <span class="note-edited">(edited)</span>
        {%- endif -%}
      </footer>
    </article>
    {%- endfor -%}
  </div>
</section>
{%- endif -%}
```

`{{ note.html | safe }}` is the ONE point where pre-sanitized server-rendered HTML
(D-04) is trusted verbatim — never re-escape it, never run it through a second
sanitizer client-side.

A new `formatDate` Nunjucks filter is needed (mirrors the existing `quantify` filter
registration in `eleventy.config.js`) — no such filter exists yet; add it alongside
`quantify`.

### Mount point for the authoring island (always present)

Below the conditional baked section, ALWAYS emit the `<bee-notes>` custom element and
its data-handoff script (mirrors the `seasonality-viz` inline-script pattern at
species-detail.njk lines 97–101) — this must render regardless of whether the baked
section above rendered anything, because a signed-in author must get the "Add note"
affordance even on a species with zero baked notes:

```nunjucks
<bee-notes id="notes-el"></bee-notes>
<script>customElements.whenDefined('bee-notes').then(function(){
  var el = document.getElementById('notes-el');
  el.canonicalName = {{ sp.canonical_name | dump | safe }};
  el.bakedNotes = {{ (notes[sp.canonical_name] or []) | dump | safe }};
});</script>
```

### How `<bee-notes>` relates to the baked `#notes` section (critical DOM contract)

`<bee-notes>` uses **light DOM** (`createRenderRoot() { return this; }`), NOT shadow
DOM — mirroring the `seasonality-viz` precedent ("seasonality-viz runs in light DOM
on taxon pages — static styles don't apply, so the SVG fill rules must live here
instead of in the component's static styles"). This is load-bearing: it lets
`bee-notes`'s own Lit-rendered markup reuse the exact same `.notes-section` /
`.note-list` / `.note` / `.note-body` / `.note-meta` classes and CSS rules as the
Nunjucks-baked markup, so the two never visually diverge.

Behavior on hydration:

1. **Default (before `fetchWhoami()` resolves, and for guests/non-authors after it
   resolves):** `<bee-notes>` renders nothing (`display: contents` or an empty
   template) — it stays fully inert. The baked `#notes` section (if present) is the
   only thing on screen. This is the offline/no-JS/guest/non-author case, and it must
   look IDENTICAL to a page that never loaded the script at all.
2. **Author, confirmed via `fetchWhoami()`:** `<bee-notes>` takes over the display —
   it renders its OWN complete `.notes-section` (heading + "Add note" button + full
   note list with per-own-note Edit/Delete, seeded from its `bakedNotes` property so
   there is no extra network round-trip on load) and hides the Nunjucks-baked `#notes`
   section underneath it (e.g. `document.getElementById('notes')?.setAttribute('hidden', '')`
   the moment it decides to render the author view). This avoids a duplicate note list
   in the accessibility tree. After any successful create/edit/delete (D-02), it
   re-fetches from the read endpoint and re-renders from that live data instead of
   `bakedNotes`.
3. **Zero baked notes, confirmed author:** there is no `#notes` element to hide (the
   Nunjucks conditional omitted it) — `<bee-notes>` renders the author empty-state
   view described in the Copywriting Contract directly into itself.

### Component states (visual hierarchy summary)

| Viewer | Notes exist? | What renders |
|--------|-------------|---------------|
| Guest / no JS / offline | Yes | Baked `#notes` section only (list, newest-first) |
| Guest / no JS / offline | No | Nothing (section omitted) |
| Non-author (signed in, not allowlisted) | Yes | Baked `#notes` section only — `<bee-notes>` confirms non-author and stays inert |
| Non-author | No | Nothing |
| Author | Yes | `<bee-notes>`'s own render: heading + Add-note button + list with Edit/Delete on the author's own notes, plain read-only display of others' notes |
| Author | No | `<bee-notes>`'s own render: heading + "No notes yet — be the first to add one." + Add-note button |

## Interaction Contract

### Add note
1. Author clicks "Add note" (`.note-btn--primary` in `.notes-heading-row`) → the
   inline `.note-editor` expands directly below the heading row (above the note
   list — newest-first ordering means new content's eventual position is at the top,
   so the editor opening at the top is spatially consistent).
2. Focus moves into the `.note-textarea`.
3. Only one editor (add OR any single edit) may be open at a time. Opening a new one
   while another is open closes/discards the previous one without a confirmation
   prompt (notes are short; this keeps the interaction simple per KISS — no
   unsaved-changes guard is in scope for this phase).
4. `Escape` while the textarea has focus cancels the editor (matches the existing
   `bee-header` popover Escape-to-dismiss convention) — draft is discarded, focus
   returns to "Add note".
5. Clicking "Post note" with an empty/whitespace-only textarea does nothing (button
   stays enabled but the field's `required`-equivalent client check blocks submit;
   no error message needed for empty submission — just don't send the request).
6. On submit: disable "Post note"/"Cancel", show "Saving…" (`.note-status`,
   `aria-live="polite"`) next to the buttons. This is **not** an optimistic update —
   wait for the API response before touching the list.
7. Success: re-fetch this species' notes from the read endpoint (D-02), re-render
   the list from that fresh data, close the editor, announce "Note posted." via the
   same `aria-live` region, return focus to "Add note".
8. Failure: re-enable the buttons, show the error copy in `.note-error`
   (`role="alert"`), keep the editor open with the author's typed text intact.

### Edit (author's own note only)
1. "Edit" button appears in `.note-owner-controls` under the author's own notes only
   (never on other authors' notes — Phase 180 curator override is out of scope).
2. Clicking "Edit" opens the same `.note-editor` shell in place of that note's
   `.note-body` (not a separate location), pre-filled with the note's raw markdown
   source (the read endpoint must supply `body_md` for the viewer's own notes so the
   textarea can be pre-filled with source, not the rendered HTML — an API-side
   dependency this UI contract assumes but does not itself specify). Submit label is
   "Save changes" instead of "Post note".
3. Same save/cancel/error/loading mechanics as Add note, above.

### Delete (author's own note only)
1. "Delete" button (`.note-btn--danger`) appears next to "Edit" in
   `.note-owner-controls`.
2. Clicking "Delete" replaces `.note-owner-controls` in place with
   `.note-delete-confirm`: "Delete this note?" + "Delete" (danger) + "Cancel" — an
   inline two-step confirmation, NOT a native `confirm()` dialog and NOT a modal
   (consistent with D-01's no-modal instruction extended to this action).
3. Clicking "Cancel" in the confirm row reverts to the normal Edit/Delete controls.
4. Clicking "Delete" (confirm) disables both buttons, shows "Deleting…"
   (`.note-status`, `aria-live="polite"`).
5. Success: re-fetch and re-render (the note disappears from the list — this is a
   soft-delete server-side per D-07, but the UI shows it simply gone), announce
   "Note deleted."
6. Failure: revert to the normal Edit/Delete controls, show the error copy inline
   (`.note-error` under the note), do not remove the note from the visible list.

### Loading / optimistic / error summary

| State | Visual treatment |
|-------|-------------------|
| In-flight (any create/edit/delete) | Buttons disabled; `.note-status` text ("Saving…"/"Deleting…") appears next to them; `aria-live="polite"` region so screen readers hear it |
| Optimistic update | **None.** Per D-02, the island re-fetches from the read endpoint after a *confirmed* success and re-renders from that response — it never speculatively inserts/removes a note before the server confirms. This avoids a whole class of "optimistic entry doesn't match what the server actually stored (e.g. rejected/truncated)" bugs for a first UGC feature. |
| Error (any action) | Inline `.note-error` (`role="alert"`, red `--error` text) near the control that failed; the editor (if open) stays open with the draft intact; the list is left as it was before the attempt |
| Divergence between live island and nightly `notes.json` | **Expected, not an error.** A note created via the island appears immediately in the live view but won't appear in the next reader's baked page until the next nightly harvest (up to 24h). Do not surface this as a warning to the author — it is invisible to them (their own view is always the live one). |

### Auth gating

`<bee-notes>` calls `fetchWhoami()` (from `src/auth-client.ts`) independently in
`connectedCallback()` — it does NOT read auth state off the `<bee-header>` DOM
element (no cross-island coupling; see RESEARCH.md Pattern 5). While the fetch is
in flight (typically well under a second), `<bee-notes>` renders nothing extra,
identical to the guest state — there is no visible "checking who you are" loading
flash.

## Accessibility Contract

- **No-JS baseline is the accessibility floor.** The baked `#notes` section is plain
  semantic HTML (`<section>`, `<h2>`, `<article>`, `<footer>`, `<time datetime>`,
  real `<a href>` for the byline) — fully legible and navigable with a screen reader
  or with CSS/JS disabled. This must never regress; it's the offline-safe read path.
- **`<time datetime="...">`** on every timestamp, with the human-readable "Jul 4,
  2026" format as the visible text (machine-readable ISO string in the attribute).
- **Byline links are real anchors**, not `<button>`+`onclick` — works with JS
  disabled, works with a screen reader's link-list navigation.
- **`aria-live="polite"`** region (`.note-status`) for all async status text
  ("Saving…", "Note posted.", etc.) so screen-reader users get non-intrusive
  announcements without a focus jump.
- **`role="alert"`** on `.note-error` so failures are announced assertively (errors
  are rarer and more important than routine status).
- **Focus management:** Add note → focus enters textarea. Cancel → focus returns to
  the button that opened the editor (Add note, or the note's own Edit button).
  Successful submit → focus returns to "Add note" (create) or stays on the note's
  position in the list, moving to its (now-closed) Edit button (edit) — never left
  on a removed/hidden element.
- **`aria-label`** on Edit/Delete buttons includes "your note" context ("Edit your
  note", "Delete your note") since a bare "Edit"/"Delete" is ambiguous when a screen
  reader user tabs through a list of several notes without also hearing which one.
- **Textarea accessible name** via `aria-label="Note text"` (see Copywriting
  Contract — no new visually-hidden-label utility introduced).
- **Focus-visible rings**: every new interactive element (`.note-btn`,
  `.note-textarea`, byline links inherit the global `a` style) gets the site's
  standard `2px solid var(--accent)` / `2px` offset ring — matches `.traits-dl dt`,
  `.node-badge`, `.species-index` control focus rings exactly.
- **Delete confirmation is keyboard-operable** inline content, not a trapped-focus
  modal — Tab reaches "Delete"/"Cancel" in the confirm row in document order, `Esc`
  is not required to dismiss it (Cancel button suffices), consistent with not
  introducing new modal-focus-trap machinery this phase doesn't otherwise need.
- **Color is never the sole signal.** The primary/danger button distinction is
  carried by both the text label ("Post note" vs "Delete") and the border/text
  color — never color alone.
