<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → service worker | The SW interposes on every fetch the `/app` page issues; a mis-scoped SW would interpose on `/` too |
| crawler → unlisted route | `/app` serves the full SPA with no access control; discoverability is the only control |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-147-01 | Tampering | `src/sw-registration.ts` import topology | mitigate | Registration imported ONLY by `src/app-entry.ts`; `_pages/index.html` + `src/bee-atlas.ts` never import it (verified by `grep -rln sw-registration src/` == 1 and `git diff --quiet` on both files). DevTools confirms no SW on `/` (checkpoint step 5). This is the central correctness+security concern of the phase. |
| T-147-02 | Information Disclosure | unlisted `/app` route (D-07) | accept | User explicitly accepted (D-07): no `noindex`/robots, relying on absence of inbound links during dogfooding; trivially reversible (add `noindex` later). Residual risk: a crawler discovers `/app` via some out-of-band link and indexes the dogfood SPA. Low severity — same public data as `/`. |
| T-147-03 | Tampering | `public/app/sw.js` URL stability | mitigate | Served from `public/app/` (Vite passthrough) so the SW URL never changes across builds; a hashed URL would defeat the browser's SW update detection. Verified by `_site/app/sw.js` existing unhashed (Task 4). |
| T-147-SC | Tampering | npm/pip/cargo installs | n/a | No package installs in this plan (RESEARCH: Package Legitimacy Audit not applicable — zero new dependencies). |
</threat_model>
