# Phase 999.1: Surface shift-drag rectangle selection in the UI - Context

**Gathered:** 2026-06-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Make the existing **shift-drag bounding-box gesture discoverable**. Today, holding Shift and dragging on the map draws a rectangle that filters occurrences (`bee-map.ts` `mousedown`+`shiftKey` → `selection-drawn` event → `_filterState.bounds`), but nothing in the UI tells users this exists.

**Purely additive discoverability — NO behavior change.** This phase adds a persistent hint that communicates the gesture. The gesture itself, the bounds-filter plumbing (Phase 999.8), and near-me (Phase 153) are unchanged. Reframed by the 2026-06-21 roadmap note: since 999.8 reclassified a spatial box as a **FILTER** (not a selection), "surfacing" means making the bounds-**filter** gesture discoverable.

</domain>

<decisions>
## Implementation Decisions

### Scope
- **D-01:** Pure **desktop** discoverability of the existing shift-drag gesture. No new drawing path, no touch box-drawing, no behavior change. (Adding a tap-to-draw mode was explicitly rejected as out-of-scope new capability.)
- **D-02:** The affordance is **hidden on touch devices** (the gesture can't fire without a Shift key). Detect via a `hover`/`pointer` capability media query (e.g. `@media (hover: hover) and (pointer: fine)`) rather than UA sniffing. Mobile users see no affordance and keep relying on near-me, exactly as today.

### Affordance form
- **D-03:** A **persistent hint text** line, reusing the existing `.hint` class in `bee-pane.ts` (muted gray, ~0.875rem — same pattern as "No sources selected…" / "Click a point on the map…"). No new UI pattern — respects [[feedback_no_unrequested_ui_patterns]].
- **D-04:** The hint is **always visible** on desktop, regardless of filter/bounds state. (Hide-once-bounds-active and hide-after-first-use were both considered and rejected — keep it simple, no persistence state.)

### Placement
- **D-05:** The hint sits **in the sidebar filters section, immediately below the "County, ecoregion, or place" (`where`) input** (`bee-pane.ts`) — NOT overlaid on the map canvas. This pairs the hint with the near-me crosshair button and where active bounds already render (Phase 153), so it appears exactly where bounds are managed.

### Copy
- **D-06:** Hint text is exactly: **"Shift-drag on map to set bounds"** (user-locked wording; aligns with the FILTER/bounds vocabulary from 999.8).

### Claude's Discretion
- Exact hint styling/spacing, optional emphasis on "Shift" (bold/`<kbd>`), and whether an icon accompanies the text — pick what reads cleanly with the existing `.hint` usages.
- Precise media-query breakpoint expression for the desktop-only / hide-on-touch gating (D-02) — choose the cleanest capability query.
- Whether the hint is rendered conditionally in the Lit template vs. always-rendered-then-CSS-hidden on touch — whichever is simplest given the `.hint` precedent and keeps it out of the DOM/flow on touch if possible.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`.hint` class** (`bee-pane.ts` ~420-424): existing muted-text pattern used for empty-state/instruction lines — reuse verbatim for the new hint (D-03). No new styling pattern needed.
- **`where` input block** (`bee-pane.ts` ~1046-1073): the anchor element; the hint renders immediately after it in the filters section (D-05).

### Established Patterns
- Pure-presenter rule: `bee-pane.ts` receives state as properties and emits events upward. A static, always-on hint needs no new state plumbing — it can be literal template content (gated by a CSS media query for desktop-only).
- Desktop-only via CSS capability query is the cleanest gate (D-02) — avoids JS device detection and respects the no-new-pattern preference.

### Integration Points
- None functional. This is a presentation-only addition adjacent to the existing `where` input; it does not touch the shift-drag handler, bounds-filter state, URL contract, or near-me path.

</code_context>

## Deferred Ideas

- **Touch / tap-to-draw bounds mode:** Letting touch users draw a bounds box (a button entering a plain-drag "draw mode") was raised and explicitly deferred — it's a new capability, not surfacing. If wanted, it belongs in its own phase. For now, mobile bounds entry stays via near-me only.
- **First-visit onboarding / dismissible hint (localStorage):** Considered for affordance persistence but rejected in favor of an always-on hint (D-04). Could revisit if the persistent hint proves noisy to repeat visitors.
