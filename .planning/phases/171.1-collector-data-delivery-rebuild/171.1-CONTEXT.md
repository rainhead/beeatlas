# Phase 171.1: Collector Data Delivery Rebuild - Context

**Gathered:** 2026-06-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Change *how* the two collector-data artifacts (`public/data/collectors.json`, `public/data/collector_event_pages.json`) are delivered to the build and to users — moving them off git tracking and onto the established S3 + `manifest.json` + `deploy.yml` fetch pattern that every other build-time data file already uses, and adding a nightly→deploy trigger so the static collector pages refresh daily.

This is delivery/plumbing only. No change to page content, the export query logic (`collectors_export.py` / `collectors_events_export.py`), or the rendered HTML. The undetermined-events bug fixed earlier this session (commits `78cffaee`, `8a305e55`) rides along once fresh data flows.

**Why this phase exists:** Phase 171 committed a 29 MB daily-regenerated artifact to git via a `.gitignore` allowlist — deviating from the `species.json` pattern, bloating history ~150 MB, and leaving live pages stale (nightly regenerates the file but never commits it back or rebuilds the site). See memory `feedback_no_committed_data_artifacts`.

**Out of scope:** any change to collector-page UI or stats; the export SQL; the Accomplishment View (Phase 172).
</domain>

<decisions>
## Implementation Decisions

### Git history handling
- **D-01:** **Full purge.** Remove all revisions of `collectors.json` + `collector_event_pages.json` from git history with `git filter-repo` (invert-paths), then force-push `main`. Reclaims ~150 MB. Accept the disruption (force-push public history; any clone must re-clone) — operator is effectively sole committer.
- **D-02:** Before the filter-repo run, **preserve the working-tree copies** of both files (they become untracked/gitignored, not deleted from disk). Drop the `.gitignore` allowlist exceptions (`!/public/data/collectors.json`, `!/public/data/collector_event_pages.json`) so `public/data/*` once again ignores them.

### Nightly → GitHub auth for the rebuild trigger
- **D-03:** **Fine-grained PAT.** nightly.sh authenticates with a fine-grained personal access token scoped to this repo only, minimal permission needed to fire the rebuild (e.g. `repository_dispatch`). Store it in a root-only file on maderas (not in the repo) and source it from `nightly.sh`; manual rotation is acceptable.
- **D-04:** Trigger mechanism is `repository_dispatch` (per the resolved freshness decision). The `deploy.yml` workflow must be extended to accept the dispatch event in addition to its current `push: branches: ['**']` trigger. nightly fires it as the final step, after the S3 upload + CloudFront invalidation.

### Clean-checkout test data
- **D-05:** **Commit a tiny fixture** (not the real file). The fixture carries the Phase 171 extended shape (`first_page_events`, `total_event_pages`, `total_event_count`, per-event fields incl. `catalog_number`/`ecdysis_id`/`inat_url`) for a small number of collectors + at least one `collector_event_pages.json` page.
- **D-06:** Fixture lives **outside `public/data/`** (e.g. a `src/tests/fixtures/` path) so `public/data/*` stays fully free of committed data — that is the whole point of the phase. `data-collectors.test.ts` (currently reads the real committed files via `readFileSync` from `public/data/`) is repointed at the fixture path. This keeps `npm test` green on a clean checkout with zero S3 access. (Planner: confirm whether any build-time assertion still needs the real file present — if so, guard it with `existsSync` + skip when absent.)

### Cutover sequencing
- **D-07:** **Data-first, one-time manual nightly.** Order: (1) run the nightly once so `collectors.json` + `collector_event_pages.json` land in S3 + `manifest.json`; (2) ship the `deploy.yml` fetch + the test-fixture repoint; (3) untrack from git (D-01/D-02). The build must never reach a state where it needs a file that is neither committed nor in S3. Mirrors `project_occurrences_contract_release_sequence` (data before code).
- **D-08:** Both files are uploaded content-hashed and registered in `manifest.json` exactly like `species.json` / `occurrences.parquet`. Planner to decide gzip-on-S3 for the 29 MB `collector_event_pages.json` (precedent: `occurrences.db` uses `_upload_hashed_gz`).

### Claude's Discretion
- Exact fixture path/filename and number of sample collectors.
- Whether `deploy.yml` also gains a `workflow_dispatch` for manual rebuilds (cheap to add alongside `repository_dispatch`).
- gzip vs plain for the large artifact on S3 (D-08).
</decisions>
