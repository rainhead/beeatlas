<objective>
Land both collector data artifacts in S3 + manifest.json BEFORE any consumer is repointed off committed files. This is step 1 of the D-07 data-before-code-before-purge sequence: the build must never reach a state where it needs a file that is neither committed nor in S3.

Extend `data/nightly.sh` (the sole nightly entry point that owns all S3 state per CLAUDE.md) with two `_upload_hashed` calls, two `manifest.json` keys, and two drift-guard classifications — a verbatim copy of the existing `species.json` / `photos.json` handling. Then a one-time operator nightly publishes both files to live S3.

Per D-08 research override: both files use `_upload_hashed` (PLAIN), NOT `_upload_hashed_gz`. A gzip-encoded S3 object fetched by `aws s3 cp` in deploy.yml downloads raw compressed bytes that `readFileSync`+`JSON.parse` cannot parse at build time; CloudFront auto-compresses JSON for browsers anyway.

`data/run.py` needs NO change — it already writes both files to EXPORT_DIR (STEPS lines 128-129). Do NOT add S3/dispatch logic to run.py (CLAUDE.md purity invariant).

Purpose: Establish the S3 delivery path so the deploy fetch (Plan 03) and the untrack (Plan 04) can safely cut over.
Output: Modified data/nightly.sh; two new live S3 objects (collectors-<hash>.json, collector_event_pages-<hash>.json) + two manifest keys after the operator nightly.
</objective>
