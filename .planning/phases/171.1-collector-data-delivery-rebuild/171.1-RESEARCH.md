### Step 2 — `nightly.sh` hashes, uploads, and assembles manifest

**`_upload_hashed` function** (`data/nightly.sh` lines 273–282):

```bash
_upload_hashed() {
    local src="$1" basename="$2"; shift 2
    local ext="${src##*.}"
    local hash; hash=$(sha256sum "$src" | awk '{print $1}' | cut -c1-12)
    local hashed_name="${basename}-${hash}.${ext}"
    aws --profile "$AWS_PROFILE" s3 cp --no-progress \
        --cache-control "public, max-age=31536000, immutable" \
        "$@" "$src" "s3://$BUCKET/data/$hashed_name" >&2
    echo "$hashed_name"
}
```

Call signature: `name=$(_upload_hashed "$EXPORT_DIR/artifact.json" "artifact-basename")`

Optional extra args (e.g., `--content-type application/json`) are passed through `"$@"`.

**`_upload_hashed_gz` function** (`data/nightly.sh` lines 287–300): gzip-compresses, adds `--content-encoding gzip`. Hash is from the **uncompressed** source so content-addressing is stable. **Used only for `occurrences.db`** (a runtime browser-fetch artifact, NOT a build-time artifact). Do not use for JSON files that `deploy.yml` fetches via `aws s3 cp` (see "gzip vs plain" section).

## Gzip vs Plain for `collector_event_pages.json` (D-08 decision)

**Recommendation: use `_upload_hashed` (plain) for BOTH collector artifacts.**

Rationale:

1. **`aws s3 cp` does not decompress Content-Encoding: gzip.** When an S3 object is stored with `Content-Encoding: gzip` (set by `_upload_hashed_gz`), `aws s3 cp` downloads the raw compressed bytes — it does not decompress. Only HTTP clients that send `Accept-Encoding: gzip` (browsers, `curl` with decode flag) get transparent decompression. Node.js `readFileSync` reads raw bytes → `JSON.parse` on compressed bytes → exception.

2. **No existing build-time artifact uses `_upload_hashed_gz`.** `occurrences.db` (the only `_upload_hashed_gz` artifact) is a runtime browser-fetch artifact — it never goes through `deploy.yml`'s `aws s3 cp` → `readFileSync` path.

3. **CloudFront auto-compresses `application/json` for browser clients.** All build-time JSON files (`species.json`, `seasonality.json`, `higher_taxa.json`) are stored plain and CloudFront compresses on the fly. The 29 MB `collector_event_pages.json` gets the same treatment for any browser/SW fetch path.

4. **S3 storage cost of 29 MB uncompressed is trivial** and the immutable URL strategy already minimizes transfer (new URL = forced refetch only when content changes).

**Verdict:** Both `collectors.json` and `collector_event_pages.json` → `_upload_hashed`. [ASSUMED based on AWS CLI behavior; HIGH confidence based on the absence of any deploy.yml precedent for gz-encoded build artifacts]

## Repository Dispatch Wiring

### PAT Scope [VERIFIED: docs.github.com/en/rest/authentication/permissions-required-for-fine-grained-personal-access-tokens]

Fine-grained PAT for `POST /repos/{owner}/{repo}/dispatches`:
- **"Contents: Read and write"** on the `beeatlas` repository only.
- No additional permissions needed.
- Create at: https://github.com/settings/tokens?type=beta

Store on maderas as a root-only file (e.g., `~/.secrets/beeatlas-github-pat`, mode 600). Source in `nightly.sh`:

```bash
GH_DISPATCH_PAT_FILE="${GH_DISPATCH_PAT_FILE:-$HOME/.secrets/beeatlas-github-pat}"
```

### `nightly.sh` dispatch call

Add as the final step in `nightly.sh`, **after** the healthcheck ping (line 359). `gh` CLI is available on the developer machine (`/opt/homebrew/bin/gh`, v2.95.0) but its availability on maderas is unverified — `curl` is safer for cron. Use `curl` with fallback comment:

```bash
# Trigger GitHub Actions deploy so collector pages refresh from today's S3 data.
echo "--- triggering repository_dispatch ---"
if [[ -f "$GH_DISPATCH_PAT_FILE" ]]; then
    GH_PAT=$(cat "$GH_DISPATCH_PAT_FILE")
    curl -fsS --retry 3 --max-time 15 \
        -X POST \
        -H "Authorization: token $GH_PAT" \
        -H "Accept: application/vnd.github.v3+json" \
        "https://api.github.com/repos/rainhead/beeatlas/dispatches" \
        -d '{"event_type":"nightly-data-updated"}' \
    && echo "  dispatch sent" \
    || echo "WARN: repository_dispatch failed (non-fatal)" >&2
else
    echo "WARN: $GH_DISPATCH_PAT_FILE not found — skipping dispatch" >&2
fi
```

The dispatch failure is non-fatal (pages stay stale, but the data is correct in S3). The next push to `main` will rebuild anyway.

### `deploy.yml` `on:` block

Extend lines 3–5 of `.github/workflows/deploy.yml`:

```yaml
on:
  push:
    branches: ['**']
  repository_dispatch:
    types: [nightly-data-updated]
  workflow_dispatch:   # optional: manual rebuild via GitHub UI
```

`workflow_dispatch` is cheap to add alongside and enables manual one-click rebuilds. Recommended per D-09 discretion.

**Note on `github.ref` for dispatch events:** When triggered by `repository_dispatch`, GitHub Actions sets `github.ref` to the default branch (`refs/heads/main`). This means the `deploy` job's `if: github.ref == 'refs/heads/main'` (line 84) evaluates true, and the full build + deploy runs as intended.

## git filter-repo Runbook (D-01)

`git-filter-repo` is available at `/opt/homebrew/bin/git-filter-repo`.

**Prerequisites:**
- All other phase code changes must be pushed and CI must be green first (D-07 step 3).
- The working tree must be clean (`git status` shows nothing).
- Run on a fresh clone (filter-repo refuses a repo with remote history divergence it doesn't own).

**Recommended procedure:**

```bash
# 1. Fresh clone into a temp directory
cd /tmp
git clone https://github.com/rainhead/beeatlas.git beeatlas-filterrepo
cd beeatlas-filterrepo

# 2. Confirm the files exist in history (should print commit SHAs)
git log --oneline -- public/data/collectors.json | head -5
git log --oneline -- public/data/collector_event_pages.json | head -5

# 3. Run filter-repo — purge both paths from all revisions
git filter-repo \
    --invert-paths \
    --path public/data/collectors.json \
    --path public/data/collector_event_pages.json

# 4. filter-repo removes the origin remote; re-add it
git remote add origin https://github.com/rainhead/beeatlas.git

# 5. Force-push (rewrites public history)
git push --force origin main

# 6. Verify: should return nothing
git log --oneline -- public/data/collectors.json
git log --oneline -- public/data/collector_event_pages.json
```

**After force-push:** The original local clone (at `~/dev/beeatlas`) now has a diverged history. Reset it:

```bash
cd ~/dev/beeatlas
git fetch origin
git reset --hard origin/main
```

**Downstream impact:** Any other clone of the repo must `git fetch` + `git reset --hard origin/main` or re-clone. As noted in D-01, the operator is effectively the sole committer, so this disruption is acceptable.

**File preservation (D-02):** The filter-repo run does NOT delete `public/data/collectors.json` or `public/data/collector_event_pages.json` from the working tree of the clone at `~/dev/beeatlas`. After `git reset --hard origin/main`, those paths are gitignored (allowlist exceptions removed) so they remain as untracked/ignored files on disk.

**Critical ordering invariant:** The gitignore exceptions MUST be dropped in the same commit as `git rm --cached`. If only the exceptions are dropped without `git rm --cached`, git will start treating the files as untracked-and-ignored but they remain in the index — confusing state. If only `git rm --cached` is done without dropping the exceptions, git will immediately re-track them on the next commit. Both must happen together.

**Deadlock risk (mirrors `project_occurrences_contract_release_sequence`):** There is no deadlock risk here because the integration gate (`test_dbt_diff`) does NOT currently diff `collectors.json` (it's not in `LOCAL_NAMES`). Adding `collectors` to `LOCAL_NAMES` in step 1 means the NEXT nightly after step 1 will pull the fresh S3 `collectors.json` as baseline and compare against the next build's output — benign.

## Common Pitfalls

### Pitfall 1: Using `_upload_hashed_gz` for a build-time JSON artifact
**What goes wrong:** `aws s3 cp` downloads compressed bytes; `readFileSync` + `JSON.parse` throws; Eleventy build fails.
**Why it happens:** `occurrences.db` uses `_upload_hashed_gz` and works fine — but it's fetched at runtime by browsers, not at build time by Node.js.
**How to avoid:** Use `_upload_hashed` for both collector files. CloudFront compresses JSON automatically.

### Pitfall 2: `git rm --cached` and `.gitignore` exception removal in separate commits
**What goes wrong:** Files re-tracked or in limbo state between commits.
**Why it happens:** `.gitignore` exceptions override the blanket `/public/data/*` ignore. Removing the override without removing from index = tracked. Removing from index without removing the override = re-tracked on next commit.
**How to avoid:** Single commit removes both allowlist lines AND runs `git rm --cached` on both files.

### Pitfall 3: Running filter-repo on the working clone
**What goes wrong:** filter-repo may refuse to run or may leave the repo in an inconsistent state. The `git reset --hard origin/main` step after force-push is needed to realign.
**Why it happens:** filter-repo detects uncommitted changes or local/remote divergence and refuses.
**How to avoid:** Always filter-repo on a fresh clone in `/tmp`, then `git reset --hard origin/main` on `~/dev/beeatlas`.

### Pitfall 4: Dispatching before S3 has the files
**What goes wrong:** `deploy.yml` fetches from S3, gets "NoSuchKey" error, build fails.
**Why it happens:** D-07 sequencing violation — trigger fired before nightly ran.
**How to avoid:** Follow D-07 strictly. Confirm S3 has both hashed files and the manifest has both keys before enabling or firing the dispatch.

### Pitfall 5: Missing LOCAL_NAMES / INTENTIONALLY_SKIPPED entries
**What goes wrong:** Nightly prints WARN for the new manifest keys every run. Noisy logs; can mask real drift alarms.
**Why it happens:** The drift-guard (lines 174–178 in nightly.sh) WARNs on every unclassified key.
**How to avoid:** Add `'collectors'` to `LOCAL_NAMES` and `'collector_event_pages'` to `INTENTIONALLY_SKIPPED` in the same PR as the upload additions.
