## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| maderas cron -> GitHub API | nightly.sh authenticates to GitHub with a fine-grained PAT to fire repository_dispatch |
| GitHub -> deploy workflow | a repository_dispatch / workflow_dispatch event triggers a build + deploy that rebuilds public pages |
| CI runner -> S3 | the build job fetches collector data from the public site bucket via OIDC |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-171.1-06 | Elevation of privilege | PAT scope | mitigate | Fine-grained PAT scoped to the single `beeatlas` repo with only "Contents: Read and write" — the minimum for `POST /dispatches` (D-03). No org access, no other repos, no extra permissions. |
| T-171.1-07 | Information disclosure | PAT secret leaking into repo/logs | mitigate | Stored root-only at `~/.secrets/beeatlas-github-pat` (mode 600) OUTSIDE the repo; sourced via a variable; passed only in the `Authorization:` header (never URL/echo); manual rotation acceptable (D-03). |
| T-171.1-08 | Denial of service | dispatch fired before S3 has the files -> CI build fails (NoSuchKey) | mitigate | D-07 ordering: depends_on Plan 01 ("published") gates this plan; the operator confirms S3 + manifest before firing (Pitfall 4). |
| T-171.1-09 | Spoofing | repository_dispatch as an unauthenticated trigger path | accept | The endpoint requires the fine-grained PAT; the worst case of a leaked token is forcing a rebuild from `main` + S3 (no secret exposure, no content injection). Blast radius = a redeploy. |

No package-manager installs in this plan — no `T-171.1-SC` supply-chain row applies.
</threat_model>
