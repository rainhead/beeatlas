## Summary

Phase 171.1 moves collector artifact delivery from committed-files-in-git to the S3 hashed-manifest pattern already used by occurrences, species, photos, and others. ## Warnings

### WR-01: `collector_event_pages.json` lacks `existsSync` guard — ENOENT on local `npm run build`

**File:** `_data/collectors.js:34-36`

**Issue:** The phase removed the `.gitignore` allowlist exception that kept `collector_event_pages.json` committed. The loader compensates with an `existsSync` guard for `collectors.json`, but the parallel branch for `collector_event_pages.json` has no guard:

```js
// collectors.json — guarded correctly:
const collectorsArray = existsSync(collectorsPath)
  ? JSON.parse(readFileSync(collectorsPath, 'utf8'))
  : (console.warn('[collectors.js] ...absent...'), []);

// collector_event_pages.json — NOT guarded:
const collectorEventPages = process.env.ELEVENTY_RUN_MODE === 'build'
  ? JSON.parse(readFileSync(join(repoRoot, 'public/data/collector_event_pages.json'), 'utf8'))
  : [];
```

When `ELEVENTY_RUN_MODE === 'build'` and the file is absent (fresh checkout, no S3 fetch), `readFileSync` throws `ENOENT` with no user-readable message. In CI the S3 fetch step would fail first and cleanly. But any developer who runs `npm run build` locally against a fresh checkout will get a raw crash rather than the graceful fallback the other guard provides.

**Fix:**
```js
const eventPagesPath = join(repoRoot, 'public/data/collector_event_pages.json');
const collectorEventPages = process.env.ELEVENTY_RUN_MODE === 'build'
  ? existsSync(eventPagesPath)
    ? JSON.parse(readFileSync(eventPagesPath, 'utf8'))
    : (console.warn('[collectors.js] collector_event_pages.json absent — returning [] (fetch from S3 for full build)'), [])
  : [];
```

## Info

### IN-01: `$(cat)` propagates non-zero exit through `set -euo pipefail` at dispatch step

**File:** `data/nightly.sh:374`

**Issue:** The PAT file guard uses `-f` (existence) but not `-r` (readability). If the file exists with wrong permissions, `GH_PAT=$(cat "$GH_DISPATCH_PAT_FILE")` propagates a non-zero exit code and `set -euo pipefail` aborts the script — after the pipeline has fully completed, the healthcheck has pinged, and the manifest is live. The exit code of `nightly.sh` would be non-zero for a non-pipeline reason.

```bash
if [[ -f "$GH_DISPATCH_PAT_FILE" ]]; then
    GH_PAT=$(cat "$GH_DISPATCH_PAT_FILE")   # <-- unhandled failure path
```

This is a theoretical risk given the PAT file is created by and for the cron user, but it violates the intent of the "non-fatal" comment on the `|| echo WARN` below.

**Fix:** Either extend the guard to also check readability, or make the read itself non-fatal:

```bash
if [[ -f "$GH_DISPATCH_PAT_FILE" && -r "$GH_DISPATCH_PAT_FILE" ]]; then
```

### IN-02: Step comment in deploy.yml says "only these three must be present" — now five are required

**File:** `.github/workflows/deploy.yml:36-40`

**Issue:** The comment for the "Fetch build-time data from S3" step still says:

> only these three must be present in the working tree for the build to succeed

Two additional files (`collectors.json`, `collector_event_pages.json`) are now also read by `_data/collectors.js` during `npm run build` and must be present. The comment also only names `_data/species.js` as the reader, omitting `_data/collectors.js`. A developer troubleshooting a build failure would be misled.

**Fix:** Update the comment to name all five required files and both Eleventy data modules.

### IN-03: PAT expanded into curl argv — visible in process listings during invocation

**File:** `data/nightly.sh:376`

**Issue:** `"Authorization: token $GH_PAT"` is shell-expanded and passed as a command-line argument to `curl`. For the duration of the curl call, the PAT token is visible in process listings (`ps aux`) to any other user or process on the same machine.

This is standard practice for server-side shell scripts and acceptable on a single-tenant nightly box, but worth noting. The token is not echoed to the log, never committed, and is file-guarded outside the repo — so the overall PAT hygiene is sound.

**Fix (optional):** Pass the header via a curl config file or environment:
```bash
# via curl --header @- approach:
echo "Authorization: token $GH_PAT" | curl --config - ...
```
Or accept the current risk given the controlled deployment environment.

## Structural Notes

The core chain is correct end-to-end:

- `_upload_hashed` calls for both collector files produce `collectors-<hash>.json` and `collector_event_pages-<hash>.json` on S3.
- Both manifest keys are added to the JSON blob.
- `LOCAL_NAMES` gets `'collectors'` for drift-gate baseline pulls; `INTENTIONALLY_SKIPPED` gets `'collector_event_pages'` (justified by size, mirrors `occurrences_db`).
- `deploy.yml` resolves both keys from the manifest via `jq` and fetches them before `npm run build`.
- `repository_dispatch` event_type `nightly-data-updated` is consistent between `nightly.sh` and `deploy.yml`.
- `.gitignore` allowlist exceptions correctly removed.
- Tests repointed to committed fixtures; `ELEVENTY_RUN_MODE` guard makes `collectorEventPages = []` under vitest, avoiding the S3 dependency.
- Fixture shape covers: Identified/Collected events, `is_reidentification` boolean/null, `is_current`, `is_pending`, `catalog_number`/`ecdysis_id` null vs typed, `inat_url`/`species_slug` mutual exclusion, page 2+ descriptor shape.
