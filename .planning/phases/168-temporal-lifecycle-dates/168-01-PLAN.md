<objective>
Surface the **identification date** into the `occurrences` mart as a single new
`id_date VARCHAR` column so a volunteer's specimen history can be built as a
two-event timeline — **Collected** (the existing `date` column, D-04) and
**Identified** (the new `id_date`) — without snapshot-diffing. `id_date` is derived
**only** for ARM 1 (ecdysis) by parsing the dirty raw `date_identified` (keep
year-only and full dates verbatim, NULL the garbage — D-06); it is `NULL` for the
other four arms (D-08/D-09). Bump the dbt contract **37→38**, add one warn-severity
singular dbt test asserting the parse never silently drops a parseable date (D-13),
validate locally with `bash data/dbt/run.sh build`, then ship the column to S3 via the
documented data-before-code release sequence (one operator-run `SKIP_INTEGRATION_GATE=1`
nightly on maderas — gated behind Phase 167's own 37-col landing, D-12).

Purpose: `id_date` is the "Identified" anchor for the per-collector event stream
(Phase 171). Shipping it now, data-before-code, means the column is live in S3 before
any TypeScript reads it — no double-gate deadlock.

Output: `id_date` column live in `occurrences.parquet` / `occurrences.db`; one new
warn-severity singular dbt test enforcing parse completeness at every `dbt build`.

DATA-LAYER ONLY — no TypeScript / frontend changes in this phase. The feed/event-stream
UI is Phase 171; provenance/status facets are Phase 170. Implements TEMP-01 and TEMP-02
per locked decisions D-01..D-13 in 168-CONTEXT.md.

**Scope reframe (operator decisions supersede ROADMAP criteria 1-2 / TEMP-01-02 where
they conflict):** `posted_date` / iNat `created_at` is DROPPED — not a column, not an
event (D-02). Collection is served by the pre-existing `date` column — do NOT add a
`collection_date` column (D-04). The only column this phase adds is `id_date` (D-05).
</objective>
