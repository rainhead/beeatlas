# 999.12 — Ecdysis download requires an authenticated session: fix design

**Drafted:** 2026-06-24 · **Status:** design / pre-plan (backlog) · **Severity:** BLOCKS NIGHTLY

## Problem (confirmed)

Ecdysis/Symbiota's bulk download endpoint now rejects anonymous requests:

```
POST https://ecdysis.org/collections/download/downloadhandler.php   (publicsearch=1, dataset 44)
→ HTTP/1.1 401 Unauthorized
   Content-Type: application/json
   Set-Cookie: PHPSESSID=…; secure; HttpOnly
   {"error":"Unauthorized access"}
```

Reproduced via `curl` 2026-06-24 with the exact params the pipeline sends. The
pipeline's Ecdysis ingestion has **always been anonymous** —
`data/ecdysis_pipeline.py:_download_zip()` POSTs `schema=symbiota … publicsearch=1`
with no login, cookie, or token. So this is an **upstream breaking change** (the
public-download path was closed / now gated), **not** an expired secret on our
side. A clean JSON 401 (not a 5xx) + a freshly-issued `PHPSESSID` is the tell that
a logged-in session is now expected.

**Blast radius:** `ecdysis` is `run.py` STEP 1, so the **entire nightly fails at the
start** every run (the cron uses a 24 h interval > the 6 h ZIP cache TTL, so it
always refetches → always 401). Production data (S3) goes stale until fixed. The
**code deploy is unaffected by this directly** — `.github/workflows/deploy.yml`
only pulls finished artifacts from S3; it never calls Ecdysis.

No clean alternative export: `https://ecdysis.org/content/dwca/` → 403.

## Approach: authenticated Symbiota session

Symbiota login form (probed 2026-06-24):
`<form name="loginForm" method="post" action="/profile/index.php">` with fields
`login`, `password`, `remember`, and a submit control (`loginButton`; Symbiota's
handler also recognizes `submitlogin` — **verify against the live form** before
implementing). A successful login binds rights to the `PHPSESSID` cookie, which
`downloadhandler.php` then honors.

Implementation sketch (in `data/ecdysis_pipeline.py`):
1. Build a `requests.Session()`.
2. `session.get("https://ecdysis.org/profile/index.php")` first if a hidden
   CSRF/anti-forgery token is present in `loginForm` (check — some Symbiota builds
   add one); include it in the login POST if so.
3. `session.post("https://ecdysis.org/profile/index.php", data={login, password, remember:0, <submit>})`.
   Assert login succeeded (no "Unauthorized"/login-form bounce; the session now
   carries an authenticated `PHPSESSID`).
4. Reuse the **same session** for the existing `downloadhandler.php` POST (move the
   current `requests.post(...)` onto `session.post(...)`).
5. Guard the response: assert `Content-Type` is the ZIP (not `application/json`);
   on a JSON `{"error":…}` body, raise with the message (so failures are loud, not
   silently cached as a corrupt ZIP).

## Credentials & secrets

- **Needs:** an Ecdysis/Symbiota account with **download rights to dataset 44**
  (WABA). [OPEN — see questions.]
- **Storage:** `data/.dlt/secrets.toml` (gitignored; mirrors how `dataset_id` lives
  in `data/.dlt/config.toml`) on maderas, e.g.
  `[sources.ecdysis] username = "…"  password = "…"`. Read via
  `dlt.secrets.value` / `os.environ`. **Never commit.**
- **CI:** no GitHub secret required — the deploy workflow does not fetch Ecdysis.
  Only maderas (the nightly host) needs the credential.

## Risks / considerations

- **Brittleness:** screen-scraping a login is fragile (form field names, CSRF
  tokens, redirects, rate limiting). Add a clear error when the download returns
  JSON instead of a ZIP so a future auth change fails loudly.
- **ToS:** confirm the account's access permits automated download of dataset 44.
- **Resilience:** keep the existing ZIP cache; consider making an Ecdysis
  login/download failure **non-fatal when a cached ZIP exists** (warn + reuse),
  so one upstream hiccup doesn't zero out a nightly. (Today a hard 401 aborts the
  whole pipeline.)
- **Lowest-maintenance alternative:** ask Ecdysis whether public download was
  intentionally removed and whether they can restore it or provide an
  API key / sanctioned DwC-A endpoint for dataset 44 — preferable to maintaining a
  login scraper if available.

## Validation (for the eventual plan)

- Unit: mock the session; assert (a) login POST is sent with the right fields,
  (b) the download reuses the authenticated session, (c) a JSON 401 body raises,
  (d) cache-fallback path still works (no network).
- Integration (maderas, manual): one real authenticated download yields a valid
  ZIP and the pipeline completes through `generate-sqlite`.

## Decoupled immediate unblock (NOT this fix — for the v5.2 deploy only)

The v5.2 deploy needs only the `occurrence_places` **schema** in S3, not fresh
Ecdysis data. On maderas (cache ZIP for dataset 44 should still be on disk from
the last good nightly):

```bash
ECDYSIS_CACHE_TTL_SECONDS=99999999 bash data/nightly.sh   # reuse cached ZIP, skip the 401
gh run rerun 28072533446 --repo rainhead/beeatlas --failed
```

If no cached ZIP exists, run the downstream-only path instead
(`bash data/dbt/run.sh build` → `uv run python data/sqlite_export.py` → regenerate
`manifest.json` + S3 push) — regenerates everything from the existing duckdb with
no Ecdysis call.
