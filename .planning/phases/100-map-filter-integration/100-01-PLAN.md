## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL → application | User-controlled URL query string (`?place=<slug>`) is parsed and fed into a SQL WHERE clause and a Mapbox feature-state lookup |
| Frontend → SQLite | `selectedPlace` value is interpolated into a SQL string in `buildFilterSQL` |
| Pipeline output → frontend | `places.geojson` properties (slug) consumed by Mapbox source and click handlers |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-100-01 | Tampering | `buildFilterSQL` in src/filter.ts — user-controlled `selectedPlace` interpolated into SQL string | mitigate | Apply single-quote doubling (`replace(/'/g, "''")`) before interpolation, mirroring the existing county clause at lines 233-236. Task 1 acceptance criterion includes a test asserting `place_slug = 'o''brien-ranch'` for an input slug containing an apostrophe. SQL is executed against a read-only in-memory SQLite (wa-sqlite WASM) — no destructive escalation possible even if escaping failed, but the test prevents quote-injection breaking the query. |
| T-100-02 | Injection (XSS) | URL `?place=<slug>` parsed and surfaced — handled in later plans, but type is established here | accept | The chip label rendering and slug storage are in Plan 02. This plan only stores the string in `FilterState`. Slug validation regex `[a-z0-9-]` is enforced at pipeline time (Phase 97 PLC-03), so production data cannot carry HTML metacharacters. User-injected URL slugs that don't match any known place will simply fail to match in the WHERE clause and produce zero rows — no exploitation surface. The chip-label XSS surface is mitigated in Plan 02 via Lit's text-binding (no `unsafeHTML`). |
| T-100-03 | Information disclosure | `places_meta` exposes `land_owner` field over public CDN | accept | `places.json` is intentionally public — owner info is published in place static pages (Phase 99). No new exposure introduced by adding the manifest entry. |
| T-100-SC | Tampering (supply chain) | npm/pip/cargo installs | mitigate | No new package installs in this plan. All work modifies existing source files. No package legitimacy audit required. |
</threat_model>
