<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| network → SW cache | The SW precaches assets fetched over the network during the first online visit; cached copies are later served offline without revalidation. |
| SW scope `/app/` → site root `/` | A SW's controllable scope is bounded by its registration scope; misconfiguration could let it intercept `/`. |
| build pipeline → `_site/app/sw.js` | The precache manifest is injected at build time; an empty/incorrect manifest changes runtime behavior. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-148-01 | Tampering | SW precache cache (serving stale/wrong app-shell assets) | mitigate | Workbox `precacheAndRoute` keys each entry by a build-time revision hash; changed assets get new hashed URLs and the old precache entry is cleaned up. `Cache-Control: no-cache` on `/app/sw.js` (Phase 147 CDK behavior, already live) forces the browser to re-check the SW on every load so a new manifest activates promptly. |
| T-148-02 | Elevation of Privilege | SW scope expansion to root `/` | mitigate | Registration is fixed to `scope: '/app/'` in `src/sw-registration.ts` (unchanged, Phase 147); a script at `/app/sw.js` has a default max scope of `/app/` and the browser rejects broader scopes. The `NavigationRoute` is allowlisted to `/^\/app\//` only (D-05) so it cannot serve the cached shell for `/` navigations and shadow the main site. Import-topology guarantee: only `src/app-entry.ts` imports the registration module, so `/`'s entry never registers a SW. `injectRegister: null` (D-06) prevents the plugin injecting a second, differently-scoped registration. |
| T-148-03 | Information Disclosure | Stale SW serving an old app version | mitigate | NO `skipWaiting` / NO `clientsClaim` (D-04) preserves the browser's atomic version transition — a new SW waits until all `/app` tabs close, so app code and (future) cached DB never present a mismatched version pair. |
| T-148-04 | Denial of Service | `self.__WB_MANIFEST` not replaced → empty precache, SW install fails | mitigate | Build-output assertion `(OFF-01, criterion 1)` fails the CI deploy gate if the literal placeholder survives or no `{url:` entry is injected; `(OFF-01, criterion 4)` fails if any precached URL is absent from `_site/`. Absolute `outDir`/`globDirectory`/`swDest` paths (D-01 discretion, RESEARCH Pitfalls 1-2) prevent the plugin writing to the deleted `.11ty-vite/` temp dir. |
| T-148-05 | Tampering | Over-precaching large `/data` artifacts into the shell cache | accept | `globIgnores` excludes `data/**`, `feeds/**`, `*.db`, `*.geojson`, `*.parquet`, `*.png`, `**/sw.js` (D-02); the 30 MB cap (D-03) is set deliberately for Phase 149's runtime DB cache, not the precache. Low risk: all `/app` chunks are <100 KB. |
| T-148-SC | Tampering | npm install of vite-plugin-pwa / workbox-* | mitigate | All five packages already declared+installed and slopcheck = OK (RESEARCH §Package Legitimacy Audit; no `[ASSUMED]`/`[SUS]`/`[SLOP]`). No new install in this plan → no legitimacy checkpoint required. |
</threat_model>
