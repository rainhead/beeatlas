The seed prints checkpoints every 50 species (e.g. `... checkpoint: 50 new entries written, 50/735 processed`). If the run is interrupted (Ctrl-C, network blip), simply re-run the same command — D-01 fill-only ensures previously-written entries are preserved and only missing species are re-fetched.

2. After completion, capture the final summary line (the one starting with `Done. processed=...`). It contains:
   - `processed`: total species iterated (should equal `species.json.length`)
   - `added`: new entries added (should equal `processed` on first run, ~0 on subsequent re-runs)
   - `skipped(existing)`: pre-existing entries (should be 0 on first run)
   - `no_taxon_id`: species without a DuckDB bridge `taxon_id` (should be 0 — RESEARCH.md confirms 735/735 coverage; non-zero indicates a regression in Phase 77's bridge population)
   - `no_photos`: species the seed found no licensed photos for (expected non-zero — checklist-only species, or species whose top observations are all-rights-reserved)

3. Validate the seeded manifest:

```bash
node scripts/validate-species.mjs
```

Expected outcome: exit 0 with output `ok content/species-photos.toml (~735 species, 0 warning(s))`.

If errors surface (license/attribution rejected by validator that the seed wrote):
- This would indicate a bug in `extractPhotos` (Plan 02 Task 1) — the seed enforces the whitelist at write time, so the validator should never see a violation. Investigate any specific error: which photo? what license_code? Most likely culprit: an iNat response field naming change.
- Fix Plan 02 if needed (RED→GREEN cycle), re-run only the affected species via incremental seed re-run (D-01 fill-only means re-running won't re-fetch already-good species — but a buggy entry would still be there; manually remove the bad table from the TOML, then re-run).

4. Run the full build chain:

```bash
npm run build
```

Expected outcome: exit 0. Build chain runs `validate-schema` → `validate-species` → `typecheck` → `eleventy`. If `validate-schema` fails, that's a Phase 78 issue (parquet schema), not Phase 79 — surface it but do not block. If `validate-species` fails, see step 3.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Live iNat API → manifest content | One-shot seed run is the moment trust crosses; once committed, manifest is in-repo and reviewable in PRs |
| Seeded manifest → CI build chain | Validator gate (Plan 01) is the trust filter |
| Wall-clock seed run → working tree integrity | Long-running process must not corrupt manifest if interrupted |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-79-14 | Tampering | iNat returns hostile photo (e.g. trolling user with offensive caption) | accept | Manifest is in git; every entry is reviewable in PRs and editable by humans (D-01 fill-only protects edits). Phase 80's renderer escapes attribution as text node (PHOTO-03). The seed cannot semantically judge content. |
| T-79-15 | DoS | Seed run exceeds iNat rate limits, API blocks BeeAtlas IP | mitigate | Plan 02's `RateLimiter` enforces ≥1 sec between requests. User-Agent identifies the project. Run is one-shot; subsequent runs are no-ops (D-01) so block risk is single-incident at most. |
| T-79-16 | Tampering | Mid-run crash leaves manifest in partially-seeded state | mitigate | Plan 02 incremental writes every 50 species; D-01 fill-only allows safe resume. Task 2 step 1 explicitly documents resumability. |
| T-79-17 | Spoofing | Manifest committed by an unauthorized party | accept | Commit signing / PR review is the project-level mitigation; out of scope for this plan. |
| T-79-18 | Repudiation | No record of which iNat snapshot the manifest was seeded against | mitigate | Task 4 commit message records timestamp + summary line + tool versions. seed-log files retained locally for forensics (gitignored to keep repo lean). |
</threat_model>
