## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-90-01 | Tampering (input) | `queryOccurrencesByBounds` in `filter.ts` | accept | Bounds originate from `map.unproject()` which returns IEEE-754 floats from the Mapbox projection. They are never user-typed strings. Interpolating them as numeric literals matches the identical pattern in `_restoreClusterSelection` (which has shipped for months without incident). ASVS V5 input validation is satisfied: numeric literal interpolation produces SQL tokens that cannot escape into a quoted string context. No additional sanitization needed. [Low severity — accept.] |
| T-90-02 | Denial of Service | SQLite bounds query | accept | The occurrences table is bounded by the WA dataset (~14k rows). A worst-case rectangle covering all of WA returns the full filtered set; this is identical in cost to a no-filter table view, which already ships. No new DoS surface introduced. [Low severity — accept.] |
| T-90-03 | Information Disclosure | Sidebar render of bounds result | accept | All occurrence rows are already publicly displayed on the map. Surfacing them in a sidebar via a rectangle gesture exposes no data that is not already client-side reachable. The occurrence dataset is public iNat + Ecdysis data. [Low severity — accept.] |
| T-90-04 | Race condition (logical, not security) | `_onSelectionDrawn` async handler vs concurrent `_filterState` mutation | mitigate | Snapshot `this._filterState` into a local `const f` BEFORE the first `await` (mirrors `_runFilterQuery` line 330). Bounds query uses `f`, not `this._filterState`, so an in-flight filter change cannot poison the result. Documented in RESEARCH.md Pitfall 2 and enforced by the explicit `const f = this._filterState;` line specified in Task 2 <action>. |
