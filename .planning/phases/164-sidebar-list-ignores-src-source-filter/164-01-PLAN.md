<objective>
Make the SQL-driven, re-querying views (sidebar list, filter-result count `_filteredRowCount`, CSV export, table view) honor the `src=` source filter that the map already respects, so the two views agree. Promote source from the standalone `<bee-atlas>._hiddenSources` UI field into `FilterState.hiddenSources` as a first-class filter dimension (the same migration Phase 156 used for `bounds`), fold it into the single shared `buildFilterSQL` predicate and `isFilterActive`, and rewire `<bee-atlas>` so a source toggle re-runs all four affected queries. Leave the map's client-side `_visibleBySource` mechanism untouched (D-03/D-04).

Purpose: Today the map hides a source while the sidebar list still shows occurrences from deselected sources — the two views disagree. Repro: `?…&pane=list&src=ecdysis,waba_sample` restricts the map but the sidebar list still includes others. The proximate bug (per RESEARCH) is that `_onSourceFilterChanged` updates `_hiddenSources` + URL but never re-runs the list/count/table queries, AND `hiddenSources` is never passed into `buildFilterSQL`.

Output: `FilterState.hiddenSources` field; a source predicate in `buildFilterSQL`; an `isFilterActive` clause; `parseParams` returning `hiddenSources` in `result.filter`; `<bee-atlas>` rewired to `_filterState.hiddenSources` as the single source of truth with all four queries re-run on source change; new unit tests proving D-01, D-05, and the `src=` round-trip.
</objective>
