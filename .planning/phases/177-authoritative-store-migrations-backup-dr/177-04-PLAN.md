<objective>
Stand up forward-only Alembic migrations against the SQLite store (STORE-02, D-03): the Alembic scaffold (`alembic.ini`, `env.py` with `render_as_batch=True`, `script.py.mako`) and the initial migration `0001_initial_schema.py` that creates `notes` + `note_revisions` matching the plan-03 models. Alembic's `alembic_version` table IS the `schema_migrations`-style ledger — no custom ledger needed.

The forward-only contract is load-bearing and mirrors the derived-vs-authoritative regime (ADR 0002): `downgrade()` raises `NotImplementedError` in the migration AND in the mako template (so every future autogenerated migration inherits the guard). This is the authoritative-data analog of the FORBIDDEN "bypass-and-rebuild" reflex — there is no upstream to rebuild from, so a downgrade that drops `notes` is unrecoverable (Pitfall 4).

Purpose: STORE-02 requires forward-only versioned migrations owned by the write layer, never by `run.py`/the nightly. This plan builds the runner + proves the pipeline never touches it.
Output: Alembic scaffold + initial migration + `test_notes_migrations.py`.

The one-time `alembic upgrade head` against the real maderas store is an operator step (plan 177-07), not run here (D-03: migrations owned by the write-layer deploy, manual in Phase 177).
</objective>
