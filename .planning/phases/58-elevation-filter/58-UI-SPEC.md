## Component Inventory

### bee-filter-controls ŌĆö elevation inputs

**Placement (from CONTEXT D-02):** Two `<input type="number">` fields appended after the token-chip field, right-aligned on the same toolbar row. They are siblings of `.search-section` within the component's shadow DOM, not nested inside the token field div.

**Markup pattern:**
```html
<div class="elev-inputs">
  <input
    type="number"
    class="elev-input"
    placeholder="Ōåæ min m"
    min="0"
    step="1"
    .value=${this._elevMin !== null ? String(this._elevMin) : ''}
    @input=${this._onElevMinInput}
    aria-label="Minimum elevation in meters"
  />
  <input
    type="number"
    class="elev-input"
    placeholder="max m"
    min="0"
    step="1"
    .value=${this._elevMax !== null ? String(this._elevMax) : ''}
    @input=${this._onElevMaxInput}
    aria-label="Maximum elevation in meters"
  />
</div>
```

**CSS (append to `bee-filter-controls` static styles):**
```css
.elev-inputs {
  display: flex;
  align-items: center;
  gap: 4px;
  flex-shrink: 0;
  margin-top: 8px; /* sm token ŌĆö aligns with token-field vertical rhythm */
}
.elev-input {
  width: 72px;
  height: 36px;
  padding: 0 0.4rem;
  border: 1px solid var(--border-input);
  border-radius: 4px;
  font-size: 0.85rem;
  color: var(--text-body);
  background: var(--surface);
  box-sizing: border-box;
  /* Remove browser spinner arrows for compact appearance */
  -moz-appearance: textfield;
}
.elev-input::-webkit-outer-spin-button,
.elev-input::-webkit-inner-spin-button { -webkit-appearance: none; margin: 0; }
.elev-input::placeholder { color: var(--text-hint); }
.elev-input:focus {
  outline: 1px solid var(--accent);
  border-color: var(--accent);
}
```

**Width:** 72px per input. This accommodates up to 4 digits (`4392`, the WA maximum) plus the unit suffix in the placeholder without overflow. Do not allow the input to grow beyond 72px.

**Spinner removal:** Browser number-input spinners are removed via CSS. The user types directly. This keeps the compact inline layout clean.

**HTML attributes:**
- `min="0"` ŌĆö WA minimum elevation; prevents negative values in browsers that validate (CONTEXT discretion resolved: apply min="0")
- `step="1"` ŌĆö integer steps only, consistent with INT16 source data (CONTEXT discretion resolved: apply step="1")
- No `max` attribute ŌĆö upper bound is left unconstrained at HTML level; SQL handles the semantics

## Interaction Contract

### Elevation input ŌåÆ filter-changed event

- **Trigger:** `input` event on either elevation field (immediate, not debounced ŌĆö CONTEXT D-13)
- **On input:** Parse with `parseInt(value, 10)`. If result is `NaN` or the field is empty, set `_elevMin`/`_elevMax` to `null`. Otherwise set to the parsed integer.
- **Dispatch:** `filter-changed` CustomEvent with a complete `FilterState` that includes current tokens merged with current `_elevMin`/`_elevMax`.

### External reset ŌåÆ input sync

- **Trigger:** `filterState` property change detected in `updated` / `willUpdate`
- **Sync:** Read `this.filterState.elevMin` and `this.filterState.elevMax`. If they differ from current `_elevMin`/`_elevMax`, update local state.
- **Guard:** Use the same feedback-loop guard as the existing token-sync pattern ŌĆö only sync when the incoming value differs from the current local state.

### Null semantics (user-visible behavior)

| State | Visible effect |
|-------|----------------|
| Both inputs empty | No elevation filter ŌĆö all records visible |
| Min only | Records with `elevation_m IS NULL` OR `elevation_m >= min` included |
| Max only | Records with `elevation_m IS NULL` OR `elevation_m <= max` included |
| Both set | Only records with `elevation_m BETWEEN min AND max` ŌĆö nulls excluded |

This means: when a user sets both bounds, iNat samples (which have `elevation_m = null`) disappear from results. This is correct per CONTEXT D-07 and ELEV-08.

### Clear filters

When `bee-atlas` dispatches an empty `FilterState` back to `bee-filter-controls` (all tokens removed, `elevMin: null`, `elevMax: null`), the `willUpdate` sync sets `_elevMin = null` and `_elevMax = null`. The bound `.value` expressions then render as `''`, clearing both inputs. No dedicated "clear" handler needed on the elevation inputs themselves.

### URL round-trip (ELEV-07)

- Encoding: `elev_min=500` / `elev_max=1500` appended to URL params when non-null; omitted when null
- Parsing: `parseInt(p.get('elev_min') ?? '') || null` pattern (matches existing year-param pattern)
- On page load with URL params: `parseParams` constructs a `FilterState` with `elevMin`/`elevMax` pre-set; `bee-filter-controls` receives this via the `filterState` property and the `willUpdate` sync populates the inputs immediately

## Accessibility Contract

- Both inputs have explicit `aria-label` attributes (`"Minimum elevation in meters"` and `"Maximum elevation in meters"`) ŌĆö placeholder text alone is not sufficient for screen readers
- `min="0"` and `step="1"` HTML attributes communicate numeric constraints to AT
- Focus ring uses `--accent` (#2c7a2c) on white/near-white background: contrast ratio 4.5:1 ŌĆö passes WCAG AA for UI components
- Spinner removal is purely cosmetic ŌĆö native `<input type="number">` semantics are preserved
- Tab order: inputs are in DOM order after the token-field; keyboard users reach them naturally via Tab after the text search input
