<objective>
Add a new, fully independent GitHub Actions workflow that runs the fast (`-m "not integration"`) Python test suite on every branch push, fails the build on any test failure, and hard-bounds the suite runtime to under 5 minutes. This is the CI capstone of the v4.8 test-suite milestone (Phases 139–142 built and greened the suite; this phase makes it visible in CI).

Purpose: Python tests are currently invisible in CI (CI is frontend-only via `deploy.yml`). This closes that gap with a build-time gate that mirrors the maderas nightly tier's fast counterpart, without coupling to the frontend deploy.

Output: `.github/workflows/python-tests.yml` plus a confirmed green end-to-end CI run on a real push.

Covers TCI-01 (CI runs the fast suite on push, fails on test failure) and TCI-02 (hard <5 min budget gate).
</objective>

<decision_notes>
- D-01: This is a SEPARATE, fully independent workflow. Do NOT edit deploy.yml. Do NOT add any `needs:` coupling between this workflow and deploy.yml. A Python-test failure must show a red check but must NOT block the frontend S3 deploy.
- D-02 / D-02a: Trigger is exactly `on: push: branches: ['**']` — NO `pull_request` event. This mirrors deploy.yml and is an INTENTIONAL, RECORDED deviation from TCI-01's literal "and pull request" wording (single-maintainer same-repo; a pushed branch IS the PR branch). The verifier should treat push-on-all-branches as satisfying TCI-01 — the missing `pull_request` event is not a gap.
- D-03: `timeout-minutes: 5` goes on the PYTEST STEP ONLY (not the job), so setup-uv / `uv sync` time is excluded from the budget. The budget bounds "the fast suite," not CI overhead.
- Discretion (apply): run `uv sync --frozen` + `uv run pytest` INLINE — do NOT call data/scripts/verify-clean-checkout.sh from CI (its worktree+strip dance is redundant on an already-clean CI checkout). Include `concurrency` (cancel-in-progress on `github.ref`) and `--durations=10` per RESEARCH recommendation (both low-risk hygiene/diagnostics).
- Pin `astral-sh/setup-uv` to SHA `fac544c07dec837d0ccb6301d7b5580bf5edae39` with a `# v8.2.0` comment; use floating `actions/checkout@v6` to match deploy.yml convention.

| Boundary | Description |

| pushed branch code → CI runner | Any branch push (including untrusted future contributors) executes this workflow on a GitHub-hosted runner. |
| CI runner → external network | The fast suite is `-m "not integration"`: it patches iNat pacing to 0, touches no S3/AWS, and makes no real network calls (verified Phases 141–142). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-143-01 | Elevation of Privilege | Workflow GITHUB_TOKEN scope | mitigate | Set `permissions: contents: read` (least privilege). The job needs only checkout; no write, no `id-token`, no secrets. A red Python run cannot reach AWS or mutate the repo. |
| T-143-02 | Tampering | Supply chain via action version float | mitigate | Pin `astral-sh/setup-uv` to commit SHA `fac544c07dec837d0ccb6301d7b5580bf5edae39` (`# v8.2.0`). `actions/checkout@v6` uses a floating major tag to match deploy.yml's house convention (T-143-02 residual accepted as repo-consistent). |
| T-143-03 | Information Disclosure | Untrusted-branch code on `branches: ['**']` | accept | The workflow exposes NO secrets to the job (no `secrets.*`, no env secrets, read-only token). Branch code runs only as deselected-to-fast-tier pytest with no network; nothing to exfiltrate. Low risk, accepted. |
| T-143-SC | Tampering | npm/pip/uv installs | mitigate | No new packages introduced (RESEARCH Package Legitimacy Audit: zero third-party packages added). `uv sync --frozen` installs only from the committed, already-audited `data/uv.lock`.
