<objective>
Add the model + serialization foundation for the "Near me" proximity filter: a `nearMe: boolean` field on `FilterState`, the bbox + pure-SQL haversine clause in `buildFilterSQL` (gated on a separately-threaded ephemeral center — never on `FilterState`), the `?near=1` URL round-trip, and the test scaffolding (Wave 0) that proves all of it.

Purpose: This is the pure, UI-free substrate the Wave-2 wiring plan (153-02) builds on. Keeping all proximity SQL and URL logic here, fully unit-tested, means the wiring plan only has to connect events to already-proven functions. It also closes the phase's named research item the way RESEARCH resolved it empirically: **pure-SQL haversine** (the wa-sqlite build has the math extension compiled in; `SELECT sin(1.0)` returns a real value), NOT the contingent JS post-filter that Success Criterion 3 names as a fallback.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GeolocationCoordinates → SQL string | The captured lat/lon (numbers from the browser geolocation API) are interpolated into a SQL string in `buildFilterSQL`. (The center is threaded by the Wave-2 plan; this plan defines the interpolation site and its guard.) |
| FilterState → URL (buildParams) | Filter state is serialized into the address bar / history / shareable links — a public surface. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-153-01 | Information Disclosure | `buildParams` / `FilterState` | mitigate | **Location-privacy (D-07):** only the boolean `near=1` is serialized — never lat/lon. Coordinates live OFF `FilterState` entirely (threaded as a separate `nearMeCenter` arg in Wave 2), so `buildParams`, which iterates `FilterState`, structurally cannot emit them. Task 1 adds a regression test asserting the serialized param string contains `near=1` and NO coordinate fragment. |
| T-153-02 | Tampering | `buildFilterSQL` proximity clause | mitigate | lat/lon are JS `number`s from `GeolocationCoordinates`, interpolated as bare numerics exactly like the existing `yearFrom`/`elevMin` clauses — no user-typed string reaches this clause. Task 2 adds an `isFinite(lat) && isFinite(lon)` guard so `NaN`/`Infinity` (which would produce malformed SQL) skip the clause instead of being interpolated. |
| T-153-SC | Tampering | npm/pip/cargo installs | accept | No packages installed this phase (RESEARCH Package Legitimacy Audit: "not applicable — no external packages"). No install task exists; nothing to gate. |
