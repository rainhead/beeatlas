<objective>
Add the near-me affordance to `<bee-pane>`, folded into the EXISTING "County, ecoregion, or place" input (D-04 — do NOT invent a new standalone chip/control; that mistake was the reason the prior design was reverted, captured in `feedback_no_unrequested_ui_patterns`). Two pieces:

1. A geolocate-icon **button**, right-aligned within the where `.input-wrap` (mirroring the existing absolutely-positioned `.input-clear`), that on click emits `near-me-requested` upward.
2. When near-me bounds are active (a new `selectionBoundsActive` property fed by `<bee-atlas>`), a removable **chip** in the existing where `.chips` row that **displays the geolocate crosshair icon** (NOT a text label — D-05, user-confirmed), whose ✕ emits `near-me-cleared` upward.

`<bee-pane>` is a pure presenter: it does NOT own `_selectionBounds`, compute boxes, or touch the URL. It only renders the affordance from a property and emits intent events. `<bee-atlas>` (plan 153-03) owns the state and binds these events.

The geolocate crosshair icon is replicated as an inline SVG in bee-pane (Claude's-Discretion in D-04/D-05: the map control's icon is a Mapbox CSS sprite, not a referenceable inline SVG, so the only practical sharing is replication — use a crosshair: a circle with four short radial ticks, matching the GeolocateControl visual). Reuse the SAME inline SVG for both the button and the chip so they read as the same feature.

Purpose: the user-facing trigger + active-state indicator for near-me, consistent with the existing where-input chip pattern.
Output: `selectionBoundsActive` property, a near-me button, a bounds chip, two new events, and bee-pane render tests.
</objective>
