<objective>
Add the curator-gated inline "Take down" control to the `<bee-notes>` island
(D-01) and the `isCurator` signal it depends on (D-03). The control appears on
every note when the signed-in user is a curator, mirrors the existing
delete-confirm interaction, and reuses the live re-fetch-after-write pattern so a
takedown drops the note from the curator's own view immediately (D-02).

Purpose: MOD-02's user-facing surface ÔÇö a curator keeps the notes surface safe
without a deploy. Per UI-SPEC this adds exactly ONE new UI element (a
`.note-btn--danger` "Take down" button with its inline confirm/pending/error
states); NO reason-input field, NO restore UI (restore is curl-only, D-07), NO
workbench (D-02). Authorization is always re-checked server-side ÔÇö `isCurator`
is a UX affordance only, never trusted for security.

Output: `AuthState.isCurator`, `takedownNote(id)` client fn, `_isCurator` getter
+ `_renderCuratorControls` (with `_takedownConfirmId`/error/pending/403 state) in
`bee-notes.ts`. No backend dependency ÔÇö the frontend unit tests mock fetch.
</objective>
