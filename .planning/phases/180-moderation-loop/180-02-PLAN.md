<objective>
Add the curator-override authz path: a fresh per-request `_is_curator_fresh`
helper (D-04/D-05) and two curator-only routes — `POST /api/notes/{id}/takedown`
(sets `status='hidden'`, appends `action='takedown'`) and
`POST /api/notes/{id}/restore` (sets `status='approved'`, appends
`action='restore'`, curl-only per D-07). This is the core gap Phase 180 closes:
the existing owner-only PATCH/DELETE routes hard-403 any non-owner, so a curator
cannot act on another author's note today.

determines authorization), MOD-03 (auditable ledger trail — attribution lives in
the `note_revisions` row only, D-08/D-10), MOD-04 (hidden excluded from read).
The owner-only routes stay untouched — the curator path is fully separate, and
the curator is takedown/restore only, never an editor of others' content (D-05).

Output: `_is_curator_fresh`, `takedown_note`, `restore_note`, plus route/authz
tests. Depends on Plan 01 (the `reason` column the ledger rows carry, D-09).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| curator browser / curl → write API | untrusted request crosses into an authz-gated state transition |
| session cookie → server identity | cookie's baked `role` is untrusted for authz; fresh allowlist is the source of truth |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-180-01 | Elevation of Privilege | `_is_curator_fresh` / takedown+restore routes | mitigate | Fresh per-request allowlist re-read via `_current_roles()`; never trust the cookie's baked `role`. A demoted curator loses power on the next request (test-locked). |
| T-180-02 | Information Disclosure | takedown/restore note lookup | mitigate | Load-before-authz: the note-specific `db_session.get` then `abort(404)` precedes any per-note branch; the identity-level curator gate runs pre-load and leaks nothing note-specific (a guessed id yields 404, not 403-before-404). Tests lock both orderings. |
| T-180-03 | Spoofing / Tampering | forged cross-site POST | mitigate | Reuses `require_author`'s Origin allow-list gate on state-changing methods (`ALLOWED_ORIGINS`) — no new CSRF surface. Cross-origin takedown/restore → 403 (test-locked). |
| T-180-04 | Tampering | stored XSS via note body | accept | takedown/restore never touch `body`/`body_html`; existing `render_note_markdown` (nh3) sanitization is unchanged. No new sanitization surface. |
| T-180-05 | Tampering | `reason` field injection / log-forging | mitigate | `reason` written via SQLAlchemy parameterized ORM insert (no SQL/shell interpolation); stripped and normalized to `None` when empty; defensive length cap at `_NOTE_BODY_MAX_LENGTH` → 400 when exceeded. |
| T-180-SC | Tampering (supply chain) | package installs | accept | No new packages (RESEARCH Package Legitimacy Audit: not applicable). |
</threat_model>
