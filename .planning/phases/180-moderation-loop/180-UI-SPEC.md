# Phase 180 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Note

This phase adds **exactly one** new UI element: a curator-gated "Take down"
button on each note rendered by the existing `<bee-notes>` island
(`src/bee-notes.ts`), plus its inline confirm/pending/error states. There is
NO new page, NO workbench, NO restore UI (restore is curl-only, D-07), and NO
reason-input field (bare button; see Copywriting Contract). Every visual
token below is an **existing** value read directly from
`src/styles/taxon-pages.css:497-587` and `src/bee-notes.ts` — this contract
is "conform to what already ships," not "introduce a new scale." This is a
hand-rolled Lit component design system; there is no shadcn/component
library in this project.

## Spacing Scale

The notes island does not use a formal 4px-multiple token scale; these are
the **actual existing rem values** used by the `.note-btn` family
(`taxon-pages.css:497-587`), which the new curator control must reuse
verbatim — no new spacing values are introduced by this phase.

| Token (existing, as used) | Value | Usage |
|-------|-------|-------|
| control gap | `0.5rem` (8px) | Gap between buttons in `.note-owner-controls` / `.note-editor-actions` / `.note-delete-confirm` — reuse for the curator control row |
| control row margin | `0.35rem` (~5.6px) | `.note-owner-controls` / `.note-meta` top margin — pre-existing exception to the 4px grid, NOT introduced by this phase, do not "fix" it here |
| button padding | `0.35rem 0.75rem` | `.note-btn` padding — reuse verbatim for "Take down" |
| editor padding | `0.75rem` (12px) | `.note-editor` box padding — not touched by this phase |

Exceptions: `0.35rem` (~5.6px) is a pre-existing non-4px value already in
production CSS (owner controls, meta row). This phase reuses it as-is; it is
not a new exception introduced here.

## Typography

The notes island uses exactly **one weight (regular/400, inherited — no
`font-weight` rule anywhere in `.note-*`)** and **two sizes**. The new
curator control introduces no new size or weight.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Button label (`.note-btn`, incl. "Take down") | 14px (`0.875rem`) | 400 (regular, inherited) | normal (unset — single-line label) |
| Meta / hint / status / error text (`.note-meta`, `.note-hint`, `.note-status`, `.note-error`, `.note-delete-confirm`) | 12px (`0.75rem`) | 400 (regular, inherited) | normal (unset — single-line text) |
| (Confirm-row prompt text, e.g. "Take down this note?") | 14px (`0.875rem`) | 400 (regular, inherited) | normal — matches `.note-delete-confirm`'s existing `font-size: 0.875rem` |

No heading/display role applies — this phase adds no headings.

## Color

There is no 60/30/10 dominant/secondary/accent surface split at the
component level — `.note-btn` is a transparent-background pill on whatever
page background surrounds it (defined by global page CSS outside this
component's scope, unchanged by this phase). The relevant existing tokens
are:

| Role | Value | Usage |
|------|-------|-------|
| Neutral button (Edit, Cancel) | `var(--text-body, #213547)` text / `var(--border, #ddd)` border | `.note-btn` base — unchanged |
| Accent (green) | `var(--accent, #2c7a2c)` | `.note-btn--primary` (Post/Save/Add note) border+text; `.note-btn:focus-visible` outline (ALL buttons, incl. the new "Take down") |
| Destructive (red) | `var(--error, #b00)` | `.note-btn--danger` border+text (owner Delete) — **reused as-is for curator "Take down"**, see Recommendation below |
| Muted / hint text | `var(--text-muted, #666)` / `var(--text-hint, #767676)` | meta row, hint, status text |

Accent (green, `--accent`) reserved for: the `.note-btn--primary` action
(Add/Post/Save note) and the universal keyboard-focus ring on every
`.note-btn` (including "Take down"). Never used for a destructive action.

**Recommendation (locked): reuse `.note-btn--danger` (red) for "Take down",
identical to owner Delete.** Color in this system encodes the *semantic
weight of the action* (destructive = red) not *actor identity*
(owner-vs-curator). Introducing a third button color for curator actions
would violate the existing convention and add a token this small phase does
not need. Distinction between "the owner's Delete" and "the curator's Take
down" is carried by **label text and `aria-label`**, not color (see
Accessibility below).

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (new curator control) | **"Take down"** (button label, mirrors "Delete" — a verb + implicit object, consistent with existing `.note-btn` labels) |
| Confirm-step prompt | **"Take down this note?"** (mirrors existing `.note-delete-confirm`'s "Delete this note?") |
| Confirm-step buttons | **"Take down"** (confirm) / **"Cancel"** (abort) — same two-button shape as the existing delete-confirm row |
| Pending-state label | **"Taking down…"** (mirrors "Deleting…") — button text swap + `aria-live="polite"` `.note-status` span reading "Taking down…" |
| Success banner (aria-live, non-error) | **"Note taken down."** (mirrors "Note deleted."/"Note updated.") — uses the existing `_banner`/`_bannerIsError=false` mechanism |
| Error state (network/5xx) | **"Couldn't take down this note. Check your connection and try again."** (new constant, same shape as existing `SAVE_ERROR_COPY`) — rendered via the existing `.note-error` style, scoped to the note via the existing per-note error slot pattern (mirrors `_deleteErrorNoteId`/`_deleteError`) |
| 403 mid-session (curator role revoked) | **"You no longer have curator permission for this action."** (new constant, sibling of the existing `OWNERSHIP_LOST_COPY`) — shown via the existing `_banner`/`_bannerIsError=true` mechanism, followed by a forced `_refetch()` so the stale "Take down" control disappears on next render |
| Empty state | **No new copy.** If a takedown empties a species' note list, the existing `.note-empty` copy — "No notes yet — be the first to add one." — already covers it; do not add a moderation-specific empty state |
| Destructive action confirmation | **Take down**: inline confirm row (not `window.confirm()`), exact shape of the existing delete-confirm interaction — see Interaction Contract below |

Explicitly OUT of scope for this phase's copy: any reason-input field/label
or restore-related UI copy (D-07 — restore is curl-only, no UI strings
needed).

## Component Contract — Curator "Take down" Control

### Visibility gate

Render the curator control block when, and only when:

```
this._authState?.authenticated === true && this._authState?.isCurator === true
```

(a new `_isCurator` getter, exact sibling shape of the existing `_isAuthor`
getter at `src/bee-notes.ts:93-95`).

- Appears on **every** rendered note, regardless of `note.can_edit` /
  ownership — a curator acts on notes they do not own (that is the entire
  point of the feature). Locked recommendation (Claude's Discretion item
  "does it appear on the curator's own notes too?"): **yes, unconditionally**
  — simplest rule, matches "alongside the owner's existing controls" (D-01)
  without a special case, and the stakes are explicitly called out as low
  either way.
- When a note is both the curator's own AND has owner controls
  (`note.can_edit === true`), both `_renderOwnerControls(note)` and the new
  curator control render as two adjacent rows (each independently uses the
  existing `.note-owner-controls` flex-row wrapper). Do not attempt to merge
  them into one button row — keep the two control families visually and
  structurally independent so ownership-vs-curation stays legible.
- The top-level island render gate (`if (!this._isAuthor) return html\`\`;`,
  `src/bee-notes.ts:318`) needs **no change** — a curator's `isAuthor` is
  always `true` server-side (`role in ("author","curator")`), so a
  curator-only login still passes the existing gate.

### Container / markup

Reuse the existing `.note-owner-controls` class verbatim for the curator
control's wrapper `<div>` — **no new CSS class or rule is needed** for the
default state:

```html
<div class="note-owner-controls">
  <button class="note-btn note-btn--danger" aria-label="Take down this note (curator)"
    @click=${() => this._openTakedownConfirm(note.id)}>Take down</button>
</div>
```

### Confirm state

Reuse `.note-delete-confirm` verbatim (no new CSS) for the inline confirm
row, swapping only copy:

```html
<div class="note-delete-confirm">
  Take down this note?
  <button class="note-btn note-btn--danger"
    @click=${() => this._confirmTakedown(note.id)} ?disabled=${this._inFlight}
  >${this._inFlight ? 'Taking down…' : 'Take down'}</button>
  <button class="note-btn" @click=${this._cancelTakedownConfirm} ?disabled=${this._inFlight}>Cancel</button>
  ${this._inFlight ? html`<span class="note-status" aria-live="polite">Taking down…</span>` : ''}
</div>
```

This requires its own state slice, separate from the owner's
`_deleteConfirmId` (a curator's takedown-confirm and an author's
delete-confirm must be independently addressable — e.g. a curator viewing
their own note could in principle have both rows open):
`_takedownConfirmId: number | null`, mirroring `_deleteConfirmId` exactly.

### Pending state

Identical mechanics to `_confirmDelete`: set `_inFlight = true`, disable both
confirm-row buttons, swap the confirm button's label to "Taking down…", show
the `aria-live="polite"` `.note-status` span. No new visual treatment.

### Post-action state (success)

`POST /api/notes/{id}/takedown` returns 200 → `await this._refetch()` (D-02,
identical to every other mutation in this component — **no optimistic
removal**). Because the read endpoint filters `status='approved'`
(unchanged, MOD-04), the taken-down note is simply absent from the next
`fetchSpeciesNotes()` response and disappears from the rendered list. Show
`_banner = 'Note taken down.'` / `_bannerIsError = false` via the existing
banner mechanism, exactly like the existing "Note deleted." flow.

### Error states

- **403** (curator role revoked mid-session, fresh-read invariant D-05):
  clear the confirm state, set
  `_banner = 'You no longer have curator permission for this action.'` /
  `_bannerIsError = true`, then `await this._refetch()` so the stale "Take
  down" control vanishes on the next render (identical shape to the existing
  `OWNERSHIP_LOST_COPY` 403 branch in `_submitEditor`/`_confirmDelete`).
- **Other failure** (network/5xx): clear the confirm state, set a per-note
  error slot + `.note-error` text — new copy, same rendering site pattern as
  `_deleteErrorNoteId`/`_deleteError` (e.g. `_takedownErrorNoteId`/
  `_takedownError`).

### Reason field: explicitly excluded from this UI (v1)

D-09 requires the **API** to accept an optional free-text `reason`; it does
not require an inline reason-input UI. **Locked recommendation:** the client
sends `reason: null` (no prompt, no textarea) — a bare "Take down" /
confirm-row interaction is sufficient for v1 and keeps the addition inside
the "one button" scope stated in the phase objective. Do not add a
`<textarea>` or `window.prompt()` for reason capture in this phase.

## Accessibility

- **Focusable / keyboard-activatable:** a native `<button>` element (as used
  throughout `.note-btn`), so Tab/Enter/Space work with no extra ARIA.
- **Focus ring:** reuses `.note-btn:focus-visible` verbatim
  (`outline: 2px solid var(--accent, #2c7a2c); outline-offset: 2px;`) — same
  accent-green ring as every other button in the island, including "Take
  down." No new focus treatment.
- **Contrast:** `var(--error, #b00)` on transparent background is the same
  red already shipped for owner Delete — no new contrast risk introduced.
- **Distinguishing curator takedown from owner delete:** carried by
  `aria-label` text, not color or position — owner Delete uses
  `aria-label="Delete your note"`; curator Take down uses
  `aria-label="Take down this note (curator)"` so a screen-reader user
  navigating a note that has both controls hears two clearly different
  actions.
- **Live regions:** reuse the existing `aria-live="polite"` `.note-status`
  span (pending state) and the existing banner's `aria-live` region (success
  / 403) — no new live-region pattern.
