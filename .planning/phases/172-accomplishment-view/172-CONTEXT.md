# Phase 172: Accomplishment View - Context

**Gathered:** 2026-06-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Add four pre-aggregated "accomplishment" displays to the **existing** per-collector
page (`/collectors/{login}/`, built in Phases 169/171):

1. A **county-coverage SVG map** ‚Äî WA counties the collector has contributed to.
2. A **taxonomic-breadth species list** ‚Äî the species they've contributed, linked to taxon pages.
3. **Ecoregion breadth** ‚Äî rendered as a **second coverage SVG map** (parallel to the county map), per the operator's twist on ACCOM-03.
4. An **"Active since YYYY (N seasons)" badge** ‚Äî derived from the collection-year range.

**Frontend + data-export only.** Everything is pre-aggregated in
`data/collectors_export.py` (extended) and a new SVG generator; the page renders
statically. The county/ecoregion data already exists on the mart (`county`,
`ecoregion_l3`) and `year`/`date` carry the collection date ‚Äî **no dbt contract
change, no new pipeline mart, no browser-side GROUP BY.**

**Out of scope:** streak tracking, leaderboards, any ranking of collectors against
each other (ACCOM-04 explicitly excludes streaks); casual-observer pages (the D-01
gate from Phase 169 stands); changes to the existing event feed / headline stats /
status split; changes to *how* `collectors.json` is delivered (that is Phase 171.1).

</domain>

<decisions>
## Implementation Decisions

### Aggregation record scope (cross-cutting ‚Äî applies to all four features) ‚Äî D-01
- **D-01:** All four aggregations (counties, species list, ecoregions, seasons) are
  computed over **WABA contributions only** ‚Äî specimens + samples, **excluding the
  collector's own casual community observations**. This is the **same row predicate
  the existing `collectors_export.py` query already uses**:
  `o.collector_inat_login IS NOT NULL AND (o.ecdysis_id IS NOT NULL OR o.record_type IN ('waba_specimen', 'provisional_sample'))`.
  Reuse it verbatim for the new aggregations ‚Äî do not invent a second scope.
  Rationale: this is an *accomplishment* view of curated collecting effort, not a
  "everywhere this person ever photographed a bee" footprint.

### County-coverage map ‚Äî ACCOM-01 (D-02)
- **D-02:** **Binary county fill, delivered as a separate per-collector `.svg` file**
  at `/data/collector-maps/{login}.svg`, generated by a new SVG step that
  **generalizes `data/species_maps.py`** (county polygon backdrop + fill). Counties
  the collector contributed to are filled one shade (reuse the existing
  `.checklist-county`-style fill class / single `<style>` block convention);
  non-contributed counties stay the blank `.county` backdrop. **No graduated
  choropleth, no per-occurrence points** (avoids locality clutter/privacy and a
  legend). Rendered on the page via `<img loading="lazy" src="/data/collector-maps/{login}.svg">`,
  exactly like `species-detail.njk` does for species maps.
- **Delivery:** these SVGs ship to S3 with **stable (non-hashed) URLs via
  `data/nightly.sh`** (recursive `aws s3 cp` + CloudFront invalidation), the
  **same mechanism already used for `species-maps/` and `place-maps/`**. They are
  **NOT committed to git** ‚Äî this sidesteps the committed-data problem Phase 171.1
  is fixing for `collectors.json`. Add a `collector-maps/` line to the nightly S3
  copy + the CloudFront invalidation paths.

### Ecoregion breadth ‚Äî ACCOM-03 (D-03)
- **D-03:** Render ecoregion breadth as a **second coverage SVG map**, parallel to
  the county map ‚Äî *operator decision: "ecoregions might as well be a map like the
  counties."* Binary fill of the **EPA Level III ecoregion polygons** the collector
  contributed to. Source geometry already exists: the `marts/ecoregions_geo` mart /
  `public/data/ecoregions.geojson` (L3 polygons clipped to WA, keyed by name). The
  SVG generator's polygon loader (`_load_county_geojsons` in `species_maps.py`) is
  **generalized to load either the county OR the ecoregion polygon set** + projects
  with the same `_project`/`_ring_to_path` helpers. Same S3 delivery as D-02
  (`/data/collector-maps/{login}-ecoregions.svg` or a parallel dir ‚Äî planner's call).
- This **supersedes** a plain named ecoregion list. Keep a short text caption with
  the **count** (e.g. "4 ecoregions") next to/under the map so "breadth" is also
  legible as a number.

### Taxonomic-breadth species list ‚Äî ACCOM-02 (D-04)
- **D-04:** **Flat species list grouped under genus headings** ‚Äî each genus is a
  small heading, its species listed beneath, **each species linked to
  `/species/{slug}/`**. Reads like the collector's personal checklist.
- **Which taxa:** **species-rank determinations only** ‚Äî the same distinct
  species-rank set already behind the existing `species_count`
  (`sp.specific_epithet IS NOT NULL`). Genus-only / coarser determinations are not
  list rows (consistent with the 169 "identified = species rank" bar, D-06).
- **Pre-computed structure:** the export writes the grouped data into
  `collectors.json` (e.g. genus ‚Üí ordered species with slug); the `.njk` template
  iterates it. Ordering: genera alphabetical, species alphabetical within genus.
- **Slug resolution:** reuse the species-slug source already used by
  `species_export.py` / the event-stream's rank-aware slug resolution (Phase 171) ‚Äî
  the mart carries only `taxon_id`; join to resolve canonical name + slug.

### Active-seasons badge ‚Äî ACCOM-04 (D-05)
- **D-05:** Badge text = **"Active since {earliest year} ({N} seasons)"** where:
  - `{earliest year}` = `MIN(year)` over the D-01 WABA-contribution rows.
  - `{N}` = **`COUNT(DISTINCT` collection year`)`** ‚Äî a "season" is a distinct field
    year with activity. **Not** a max‚àímin+1 span; gap years do not count. **No
    streaks.**
  - Source column is **`year`** (or the year of `date`) ‚Äî **NOT `id_date`**, and
    **NOT a `collection_date` column** (none exists; Phase 168 narrowed to the
    pre-existing `date`/`year`). The ROADMAP/criterion wording "collection_date
    column range" is stale ‚Äî use `year`/`date`.

### Pre-aggregation guarantee ‚Äî ACCOM criterion 5 (D-06)
- **D-06:** Every aggregation is computed in the **pipeline** ‚Äî
  `collectors_export.py` writes the badge, ecoregion count, and grouped species
  list into `collectors.json`; the two coverage SVGs are pre-generated files. The
  collector page runs **no wa-sqlite GROUP BY in the browser**. The export + SVG
  generator read `occurrences.parquet` from **`EXPORT_DIR`** (the dbt-build copy),
  NOT the dbt sandbox (Pitfall 5, per `species_maps.py` / `collectors_export.py`).

### Claude's Discretion
- Exact file layout for the two SVGs (two files vs one with two `<g>` layers; dir
  naming `collector-maps/`), the SVG generator module name, and whether to fold the
  generator into the existing `species_maps.py` machinery or a sibling module.
- Whether per-species counts appear in the taxonomic list (lean: yes, small).
- On-page placement/order of the four elements relative to the existing headline
  stats + event feed, and the section CSS (reuse `places.css`/`collectors` styles
  or extend) ‚Äî a UI-SPEC concern.
- Exact SQL shape of the new aggregations, provided the D-01 row predicate, D-04
  species-rank filter, and D-05 distinct-year semantics hold.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `collectors_export.py` `_QUERY` ‚Äî already gated to WABA contributions (D-01 row
  predicate) and already computes `species_count` as distinct species-rank taxa;
  the new aggregations are additional SELECT columns / a companion query on the
  same row set.
- `species_maps.py` is a near-complete SVG county-map generator: county polygon
  loader, linear lon/lat‚ÜíviewBox projection, ring‚Üípath, single-`<style>`-block
  classing, and a `.checklist-county` **county-fill** class that is exactly the
  binary-fill primitive D-02 needs. Generalize its polygon loader to also accept
  the ecoregion polygon set (D-03).
- `quantify` filter for all the count-noun copy.

### Established Patterns
- Static per-entity SVG maps are delivered as **stable-URL S3 files** (`species-maps/`,
  `place-maps/`) via `nightly.sh` recursive `s3 cp` + CloudFront invalidation ‚Äî
  **not committed to git, not in `manifest.json`**. `collector-maps/` follows this
  exactly (D-02). This is a different delivery path from `collectors.json` (which is
  the Phase 171.1 manifest/fetch path).
- Export steps read the **`EXPORT_DIR` parquet** copy, never the dbt sandbox
  (Pitfall 5).
- `_data` loader reads only the exported `.json` (Pitfall #8) ‚Äî the grouped species
  list + badge + ecoregion count must be in `collectors.json`, not computed at
  Eleventy time.

### Integration Points
- New: `public/data/collector-maps/*.svg` (S3-served, stable URL) ‚Äî referenced by
  `<img>` in `collector-detail.njk`.
- Extended `collectors.json` shape (badge fields, ecoregion count, grouped species
  list) ‚Äî coordinate with Phase 171.1's fixture (171.1 D-05/D-06).
- `run.py` STEPS gains the SVG-generation step; `nightly.sh` gains the
  `collector-maps/` S3 copy + invalidation path.
- No dbt/contract change ‚Äî downstream of the mart.

### Live data signals (from Phase 169)
- D-01 gate ‚âà **124 collector pages**; `county`/`ecoregion_l3` are populated on
  every occurrence row by the `occurrences.sql` spatial join + nearest-fallback
  (no NULL-county gaps to special-case).

<specifics>
## Specific Ideas

- Operator twist on ACCOM-03: *"ecoregions might as well be a map like the
  counties"* ‚Äî the load-bearing reason ecoregion breadth is a second SVG map (D-03)
  rather than a text list. The symmetry (two parallel coverage maps) is the intended
  look.
- Accomplishment framing means the maps/list/badge reflect **curated collecting
  effort**, not casual sightings ‚Äî the reason D-01 reuses the existing WABA-only
  export predicate rather than widening to all rows.

</specifics>
