<objective>
Build the in-app install affordance: capture `beforeinstallprompt`/`appinstalled` in a new
early window-level side-effect module, relay installability state through `<bee-atlas>`
`@state` to a new "Install" button in `<bee-header>`, and (on iOS Safari) open an
Add-to-Home-Screen popover that reuses the existing cache-popover machinery. This greens the
Wave 0 `install-affordance.test.ts` source assertions from Plan 01.

Purpose: PWA-01 needs the Android `beforeinstallprompt` captured (preventDefault + stashed)
and surfaced as a quiet in-app "Install" button (not a modal/banner); PWA-02 needs the iOS
Safari "Add to Home Screen" instructions, shown only when not already standalone. Both must
respect the Architecture Invariant (state lives on `<bee-atlas>`; `<bee-header>` is a pure
presenter) and the quiet-chrome principle (button absent unless actionable).

Output: new `src/install-prompt.ts`; modified `src/app-entry.ts`, `src/bee-atlas.ts`,
`src/bee-header.ts`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser event ↔ install-prompt.ts | `beforeinstallprompt`/`appinstalled` are browser-fired DOM events with no untrusted payload; the module only calls `preventDefault()` and `prompt()` on the OS-supplied event. |
| <bee-atlas> ↔ <bee-header> | State flows down as properties, user intent flows up as a `composed` CustomEvent — the established trusted in-app component boundary. No external input. |
| navigator UA / standalone read | iOS detection reads `navigator.userAgent`/`platform`/`maxTouchPoints`/`standalone` — client-controllable, but used ONLY to decide whether to show a non-privileged Install button. Spoofing it merely shows/hides a button; no security boundary depends on it. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-151-05 | Spoofing | UA-based iOS detection | accept | A spoofed UA only toggles a cosmetic Install button (no privilege, no data access). D-12 already scopes detection as best-effort; no security decision rides on it. |
| T-151-06 | Tampering | `window.__pwaPrompt` global handoff | accept | Same trust model as the existing `window.__wb` handoff (Plan 150). A function reference on `window` in a same-origin static app; any script that could overwrite it already runs in-origin. No new boundary. |
| T-151-07 | Elevation | install logic leaking onto `/` | mitigate | `install-prompt.ts` is imported ONLY from `app-entry.ts` (asserted by the module doc comment + the no-import-on-/ structural guarantee); the `/` entry (`bee-atlas.ts` via `_pages/index.html`) never imports it. Preserves no-PWA-on-/ (D-04). |
| T-151-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this plan (RESEARCH §Package Legitimacy Audit: N/A). All code reuses already-present `lit`/`workbox-window`. No supply-chain surface. |
</threat_model>
