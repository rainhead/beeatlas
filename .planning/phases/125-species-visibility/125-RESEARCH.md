# Phase 125: Species Visibility - Research

**Researched:** 2026-05-29
**Domain:** dbt SQL, Python data pipeline, Eleventy static site generation
**Confidence:** HIGH

## Summary

Phase 125 makes all species with `occurrence_count > 0` visible in `species.json`, the
species tree, and the static site — regardless of WA checklist membership. The root cause
is a single missing derivation in `int_species_universe.sql`: `specific_epithet` is only
populated when a species appears in `stg_checklist__species`; off-checklist species always
have `specific_epithet = NULL` even when their two-token canonical name encodes the epithet
directly.

At research time, 65 two-token off-checklist species have `occurrence_count > 0` and
`specific_epithet = NULL`. These are the newly-visible species targeted by SPV-01/02/03.
An additional 37 single-token (genus-level) off-checklist species also have occurrences
but cannot get species pages because they have no epithet to derive — they correctly remain
as "Genus sp." summary entries and are unchanged by this phase.

The fix is surgical: one SQL change to `int_species_universe.sql` adding a COALESCE
derivation. All downstream consumers (species_export.py, species_maps.py, species-detail.njk,
_data/species.js) already handle species with `specific_epithet IS NOT NULL` correctly and
require no changes. SVG maps are automatically generated by `species_maps.py` (its existing
filter `WHERE ... AND specific_epithet IS NOT NULL` will pick them up), and nightly.sh
already uploads species-maps recursively to S3.

**Primary recommendation:** Change two COALESCE expressions in `int_species_universe.sql`
(specific_epithet derivation and scientificName capitalization), re-run `bash data/dbt/run.sh build`,
then verify 65 new species appear with correct slugs and SVGs.

### Pattern 1: COALESCE Derivation of specific_epithet

**What:** Derive `specific_epithet` from the second token of `canonical_name` when the
species is not on the checklist. Single-token names (genus-level) get NULL (correct behavior).

**Exact change to `int_species_universe.sql` line 92:**

```sql
-- BEFORE (only checklist species get specific_epithet):
c.specific_epithet AS specific_epithet,

-- AFTER (off-checklist two-token species also get specific_epithet):
COALESCE(
    c.specific_epithet,
    NULLIF(split_part(COALESCE(c.canonical_name, oa.canonical_name), ' ', 2), '')
) AS specific_epithet,
```

**Verification (DuckDB):**
- `'halictus rubicundus'` → `split_part(..., ' ', 2)` = `'rubicundus'` (correct)
- `'lasioglossum'` → `split_part(..., ' ', 2)` = `''` → `NULLIF(..., '')` = `NULL` (correct for genus-level)
- `'Andrena milwaukeensis Viereck'` → checklist row, `c.specific_epithet = 'milwaukeensis'` (COALESCE first arm wins)

[VERIFIED: direct DuckDB expression testing — all three cases produce correct output]

### Pattern 2: COALESCE Capitalization of scientificName

**What:** For off-checklist species, `scientificName` falls through to `oa.canonical_name`
which is always lowercase (e.g., `'halictus rubicundus'`). Capitalize the first letter for
display purposes.

**Exact change to `int_species_universe.sql` line 79:**

```sql
-- BEFORE:
COALESCE(c.scientificName, oa.canonical_name) AS scientificName,

-- AFTER:
COALESCE(
    c.scientificName,
    upper(left(COALESCE(c.canonical_name, oa.canonical_name), 1)) ||
    substring(COALESCE(c.canonical_name, oa.canonical_name), 2)
) AS scientificName,
```

**Result:** `'halictus rubicundus'` → `'Halictus rubicundus'` on the species page `<h1>` tag.

[VERIFIED: direct DuckDB expression testing]

### Anti-Patterns to Avoid

- **Deriving specific_epithet in Python (species_export.py) instead of SQL:** The SQL
  layer is the correct place — `specific_epithet` flows into multiple dbt models
  (checklist mart, species mart). Putting the derivation in Python would create
  inconsistency: checklist.parquet would still not have it, and the test for
  `test_checklist_no_null_specific_epithet` (which only applies to checklist rows) would
  be unaffected. Fix once in the SQL model.

- **Using ARRAY_LENGTH(STRING_SPLIT()) to detect two-token names:** Not needed.
  `NULLIF(split_part(cn, ' ', 2), '')` correctly returns NULL for single-token names
  without requiring a token count predicate.

- **Changing the dbt `species` mart schema:** The `specific_epithet` column is already
  `VARCHAR` (nullable) in `schema.yml`. No schema contract change is needed.

- **Touching species_maps.py:** It already filters `AND specific_epithet IS NOT NULL` —
  the 65 new species will appear automatically.

## Common Pitfalls

### Pitfall 1: scientificName Remains Lowercase Without Fix

**What goes wrong:** SPV-01 only mentions `specific_epithet`. If the planner omits
the scientificName capitalization fix, species pages will show `<h1><em>halictus rubicundus</em></h1>` —
lowercase, not a proper scientific name.

**Why it happens:** `int_species_universe.sql` line 79 falls through to `oa.canonical_name`
for off-checklist species; canonical names are always stored lowercase.

**How to avoid:** Fix both `specific_epithet` and `scientificName` in the same SQL change.

**Warning signs:** After `dbt build`, inspect `sandbox/species.parquet` — off-checklist
species should have capitalized `scientificName` like `'Halictus rubicundus'`, not
`'halictus rubicundus'`.

### Pitfall 2: test_dbt_diff.py "Verified baseline" Comment is Stale

**What goes wrong:** `test_species_parquet_row_count_matches` has a docstring comment
`"Verified baseline: both 629 rows."` This is still true after the fix (total row count
is unchanged at 629). However, the test asserting sandbox == public row counts will PASS
correctly at the time the developer runs it against the local sandbox — because the public
species.parquet (at `public/data/species.parquet`) is only updated after a full pipeline run.

**How to avoid:** The test comment is descriptive, not assertive — no update needed to
the test logic. The test will pass during development because sandbox and public counts
match (both 629). After a full pipeline run, both become 629 still.

**Note:** The `specific_epithet` values in `public/data/species.parquet` will differ from
sandbox until the nightly pipeline runs — this is expected and not tested by the diff harness.

### Pitfall 3: Slug Conflict Check (Confirmed None Exist)

**What goes wrong:** A newly-derived slug like `Megachile/apicalis` could theoretically
collide with an existing checklist species slug. This would cause Eleventy to produce
two pages at the same URL, with one overwriting the other.

**How to avoid:** Verified at research time: zero ANTI-JOIN matches between derived
off-checklist slugs and existing checklist slugs. No conflicts exist.

[VERIFIED: DuckDB ANTI-JOIN query — 0 conflicts found]

### Pitfall 4: Genus-Level Off-Checklist Species (37 Records)

**What goes wrong:** 37 single-token off-checklist species (e.g., `'lasioglossum'`,
`'andrena'`) have `occurrence_count > 0` but CANNOT get species pages — they have no
`specific_epithet` to derive. SPV-01 says "with a valid specific_epithet" which these
cannot satisfy.

**How to avoid:** This is correct behavior. These 37 species remain as "Genus sp." grey
dot entries in the genus maps and genus page species list. The requirement is satisfied
for the 65 two-token species. Do not attempt to create pages for genus-level records.

**Warning signs:** If a test asserts all 102 off-checklist occurrence-bearing species
get `specific_epithet`, it will fail (correctly, for the 37 genus-level ones).
Tests should assert 65 new species gain `specific_epithet`, not 102.

### Pitfall 5: int_species_universe DISTINCT ON Order Stability

**What goes wrong:** The final `SELECT DISTINCT ON (canonical_name)` in `int_species_universe`
uses `ORDER BY canonical_name, on_checklist DESC` to prefer checklist rows when duplicates
exist. The new COALESCE for `specific_epithet` must not introduce new duplicate rows.

**How to avoid:** The COALESCE is applied within the `species_universe` CTE before the
`DISTINCT ON`, replacing NULL with derived values. It does not change the row count or
create duplicates — it fills in values on existing rows.

### Pitfall 6: inat_obs canonical names may have new specific_epithet too

**What goes wrong:** After Phase 124, `resolve_taxon_ids.py` now covers inat_obs canonical
names. Some inat_obs-derived species may appear in `int_species_occurrences_agg` with
two-token canonical names not on the checklist. The fix handles these identically to
ecdysis-derived species — the COALESCE applies to `COALESCE(c.canonical_name, oa.canonical_name)`
which covers both sources.

**How to avoid:** No special handling needed. The COALESCE pattern applies universally
to any off-checklist species with a two-token canonical name, regardless of source.

### Regression Test Pattern for SPV-01 (New Test)

```python
# Source: data/tests/test_dbt_scaffold.py (new test to add)
@pytest.mark.skipif(
    not (SANDBOX / "species.parquet").exists(),
    reason="run `bash data/dbt/run.sh build` first"
)
def test_off_checklist_species_with_occurrences_have_specific_epithet():
    """All two-token off-checklist species with occurrence_count > 0 have specific_epithet (SPV-01)."""
    parquet_path = str(SANDBOX / "species.parquet")
    n = duckdb.execute(
        f"SELECT COUNT(*) FROM read_parquet('{parquet_path}') "
        "WHERE occurrence_count > 0 AND on_checklist = false "
        "AND ARRAY_LENGTH(STRING_SPLIT(canonical_name, ' ')) = 2 "
        "AND specific_epithet IS NULL"
    ).fetchone()[0]
    assert n == 0, (
        f"Expected 0 two-token off-checklist species with occurrences to lack specific_epithet, "
        f"got {n}. Fix COALESCE derivation in int_species_universe.sql."
    )


def test_off_checklist_species_scientificname_capitalized():
    """Off-checklist species with two-token canonical names have capitalized scientificName (SPV-01)."""
    parquet_path = str(SANDBOX / "species.parquet")
    n = duckdb.execute(
        f"SELECT COUNT(*) FROM read_parquet('{parquet_path}') "
        "WHERE occurrence_count > 0 AND on_checklist = false "
        "AND ARRAY_LENGTH(STRING_SPLIT(canonical_name, ' ')) = 2 "
        "AND scientificName != upper(left(scientificName, 1)) || substring(scientificName, 2)"
    ).fetchone()[0]
    assert n == 0, (
        f"Expected 0 off-checklist species with lowercase scientificName, got {n}."
    )
```
