| Property | Value |

|------|------|--------|-------------|

- New chip text: 14px (0.85rem) at weight 400. Chip-type badge: 12px (0.75rem) at weight 500.
- Do not introduce new font sizes outside this table.

## Component Inventory

### 1. Boundary Mode Toggle (new)

Placement: Top of sidebar, above all filter controls. Renders in both `layerMode === 'specimens'` and `layerMode === 'samples'`.

Structure: Reuses `.layer-toggle` + `.toggle-btn` + `.toggle-btn.active` classes exactly. Three buttons: Off, Counties, Ecoregions.

```
[Off]  [Counties]  [Ecoregions]
```

- Exactly one button is active at all times (driven by `boundaryMode` @property from bee-map.ts).
- Active button: color #2c7a2c, border-bottom 2px solid #2c7a2c, weight 600.
- Inactive button: color #888, weight 500, hover background #f5f5f5.
- On click: dispatch `filter-changed` event with updated `boundaryMode` field (see architecture pattern).

### 2. Region Autocomplete Inputs (new)

Two inputs, one for counties, one for ecoregions. Both render below existing taxon/date controls in `_renderFilterControls()` for specimen mode, and in a separate `_renderRegionControls()` that is always rendered (both modes).

Layout recommendation (Claude's Discretion, from RESEARCH.md):
- Region controls are extracted into `_renderRegionControls()`, rendered outside the `layerMode === 'specimens'` gate — always visible below the main filter section.
- County input appears first, ecoregion input second.
- Each input uses the existing `.filter-row` class and the exact same `<input type="text" list="...">` + `<datalist>` pattern as the taxon autocomplete.

Input placeholder text:
- County: "Filter by county…"
- Ecoregion: "Filter by ecoregion…"

Selection behavior:
- On exact datalist option match: add to `@state` Set, clear input value.
- Already-selected items are not re-added (check Set membership before dispatch).
- Use `@input` + `@change` dual-handler (matching existing taxon pattern).

### 3. Region Chips (new)

Placement: Below the autocomplete inputs, within the same region controls section. Grouped by type: county chips first, then ecoregion chips.

Chip anatomy:
```
[ [county] King  ✕ ]   [ [county] Pierce  ✕ ]   [ [ecoregion] Cascades  ✕ ]
```

Type badge visibility: Show `[county]` / `[ecoregion]` badge on every chip when BOTH `_selectedCounties.size > 0` AND `_selectedEcoregions.size > 0`. When only one type is active, omit the badge (the context is unambiguous).

CSS for new elements:
```css

  display: flex;
  flex-wrap: wrap;
  gap: 0.25rem;   /* 4px xs token */
  margin-top: 0.5rem; /* 8px sm token */
}
.chip {
  display: inline-flex;
  align-items: center;
  gap: 0.25rem;   /* 4px */
  padding: 0.2rem 0.5rem; /* 3.2px / 8px */
  background: #f0f0f0;
  border: 1px solid #ccc;
  border-radius: 4px;
  font-size: 0.85rem;
  color: #333;
}
.chip-type {
  font-size: 0.75rem;
  font-weight: 500;
  background: #e0e0e0;
  color: #555;
  border-radius: 3px;
  padding: 0 0.25rem;
}
.chip-remove {
  background: none;
  border: none;
  cursor: pointer;
  color: #666;
  font-size: 0.85rem;
  padding: 0.1rem 0.2rem;

  color: #333;
```

Remove action: On click, remove item from `@state` Set, dispatch `filter-changed` event.

## Interaction Contract

### Boundary Toggle
| State | Visual |
|-------|--------|
| Off (default) | "Off" button active (green bottom-border + text) |
| Counties | "Counties" button active |
| Ecoregions | "Ecoregions" button active |

- Clicking the already-active button is a no-op (do not dispatch).
- After "Clear filters": boundary mode resets to Off and toggle reflects this.

### Autocomplete Add
1. User types in county/ecoregion input.
2. Browser shows native datalist suggestions.
3. User selects an option (exact match required — no partial adds).
4. Chip appears immediately; input clears to empty string.
5. `filter-changed` event dispatched with updated Sets.

### Chip Remove
1. User clicks ✕ on chip.
2. Chip disappears immediately.
3. If removed chip was last of its type and boundary mode was set to that type, boundary mode does NOT auto-reset (user must use the toggle or Clear filters).
4. `filter-changed` event dispatched with updated Sets.

### Clear Filters Extension
- Resets `_selectedCounties = new Set()`, `_selectedEcoregions = new Set()`.
- Resets `boundaryMode` to `'off'` in the dispatched event.
- All chips disappear; boundary toggle shows "Off" active.
- Map position unchanged (as per FILTER-06 / Out of Scope table in REQUIREMENTS.md).

## Layout Structure

```
bee-sidebar (flex column, overflow-y: auto)
├── Boundary Toggle        ← new, always visible
│   [Off]  [Counties]  [Ecoregions]
├── Layer Toggle (existing)
│   [Specimens]  [Samples]
├── Filter Controls (existing, specimen mode only)
│   ├── Taxon autocomplete
│   ├── Year range
│   └── Month grid
├── Region Controls        ← new, always visible (both modes)
│   ├── County autocomplete input
│   ├── County chips
│   ├── Ecoregion autocomplete input
│   └── Ecoregion chips
│   └── [Clear filters] button (extended)
└── Detail panel / summary (existing)
```

Note: The "Clear filters" button currently lives inside filter-controls. With region controls always visible, move the Clear filters button to the bottom of region controls so it covers both filter blocks. Alternatively, keep it in filter-controls and add a separate "Clear regions" affordance — but a single "Clear filters" covering all filters (taxon + date + region) is simpler and matches FILTER-06 intent. Use a single Clear filters button.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Boundary toggle — Off | "Off" |
| Boundary toggle — Counties | "Counties" |
| Boundary toggle — Ecoregions | "Ecoregions" |
| County autocomplete placeholder | "Filter by county…" |
| Ecoregion autocomplete placeholder | "Filter by ecoregion…" |
| Chip remove button aria-label | "Remove {name}" (e.g. "Remove King") |
| Clear filters button | "Clear filters" (existing label — unchanged) |
| Empty state (no chips) | No visible empty state message — inputs are always shown; absence of chips is self-explanatory |
| Error state | No error states — datalist input only accepts values from the provided list; invalid input is simply ignored on change |
| Destructive confirmation | Not applicable — chip remove is immediate (no confirmation); the action is trivially reversible by re-adding |

No new heading text needed. The region controls section does not require an `<h3>` heading unless the layout evaluation during implementation reveals ambiguity — if added, use "Region" (single word, lowercase, matching existing filter-controls h3 style).
