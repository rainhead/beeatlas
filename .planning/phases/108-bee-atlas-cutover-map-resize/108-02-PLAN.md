<objective>
Verify in a real browser that the Phase 108 cutover satisfies MAP-01: the Mapbox canvas resizes correctly (i.e., shows no grey tiles, misaligned controls, or stale tile edges) after every pane state transition — collapsed↔list, list↔table, and table→list→collapsed sequences — on both desktop and mobile viewports. This plan exists separately from 108-01 because the verification requires a human at a real browser; npm test cannot exercise Mapbox canvas resize behavior.

Purpose: bee-pane is `position:absolute` overlay, which means bee-map.mapElement's CSS dimensions do not change across pane transitions. The bee-map.ts ResizeObserver (line 807) calls `this._map?.resize()` whenever mapElement resizes. Because mapElement size is invariant under pane transitions, the observer does NOT fire on those transitions — and it should not need to. This plan confirms that assumption empirically: that no resize call is needed and that the map renders cleanly across all pane state changes.

Output: A signed-off browser UAT checklist documenting MAP-01 satisfaction (or, if a regression is found, a precise reproduction recipe to feed into a gap-closure plan).
</objective>

  <what-built>
    Phase 108 Plan 01 replaced the bee-filter-panel + bee-sidebar + bee-table sibling render path in bee-atlas with a single <bee-pane> overlay. The four pane navigation events (pane-expand-list, pane-collapse, pane-expand-table, pane-shrink-list) are wired to bee-atlas handlers that update _paneState and call _replaceUrlState. No explicit map.resize() call was added — the design relies on bee-pane being `position:absolute` overlay so bee-map.mapElement's size never changes across pane state transitions, leaving the existing bee-map ResizeObserver (line 807) to handle MAP-01 implicitly.
  </what-built>
  <how-to-verify>
    Prerequisites:
    1. Run `npm run dev` and wait for "Local: http://localhost:..." to print.
    2. Open the printed URL in a desktop browser (Chrome or Safari preferred — they expose Mapbox WebGL artifacts most reliably).
    3. Open browser devtools → Network tab → enable "Disable cache" (so tile reloads are observable). Keep devtools docked to the side or bottom for the desktop run; close it for the mobile-emulation run so the viewport switch is honored.

    Desktop sequence (perform each transition, then inspect the map for ~2 seconds before proceeding):

    A. Start in collapsed state (default on fresh load).
       - Verify: pane is a thin toggle strip at the right edge; map fills the rest of the content area; no grey tiles; Mapbox attribution + zoom controls visible.

    B. Click the toggle button → pane expands to list state (25rem width).
       - Verify: map canvas shrinks horizontally to make room for the 25rem pane; no grey tiles appear at the seam between map and pane; Mapbox zoom controls (bottom-right of map) reposition cleanly; attribution remains legible.
       - Pan the map slightly to confirm tiles fetch and render without stale edges.

    C. Click the expand button (inside the pane's list state header) → pane expands to table state (full-width overlay covering the map area).
       - Verify: pane covers the full content area; table renders with rows; the map is occluded but should still be present in the DOM (devtools → Elements → confirm `<bee-map>` is still rendered).
       - Note: in the overlay architecture the map's container dimensions do NOT change here, but the pane visually covers it. Confirm that on Inspect the bee-map element bounding box is unchanged from state B.

    D. Click the shrink button (inside the pane's table state header) → pane returns to list state.
       - Verify: pane shrinks back to 25rem; map is fully revealed; ZERO grey tiles, ZERO stale tile edges, ZERO misaligned Mapbox controls.
       - Pan the map slightly to confirm tile fetch resumes cleanly.

    E. Click the toggle button → pane collapses.
       - Verify: pane returns to thin toggle strip; map expands to fill content area; no grey tiles or stale edges.

    F. Run the full E→B→D→B→E cycle (collapsed → list → table → list → collapsed) three times in succession at a normal click cadence. After each transition, look for resize artifacts. Then run it again with rapid clicks (toggle, expand, shrink, toggle within ~1 second).
       - Verify: no artifacts appear under either cadence.

    G. URL bar inspection during the cycle:
       - In collapsed state, URL `pane=` param is absent or `collapsed`.
       - In list state, URL contains `pane=list`.
       - In table state, URL contains `pane=table`.
       - Confirm transitions update the URL within ~100ms of the click.

    Mobile-emulation sequence:

    H. Open devtools → Toggle device toolbar → select "iPhone 14 Pro" (or any viewport where max-aspect-ratio:1 holds, i.e. portrait orientation). Refresh.
       - Verify: pane occupies the full viewport width (right:0, left:0 per UI-SPEC mobile rules); no expand button visible (PANE-06 contract preserved).

    I. Tap the toggle button → pane opens to list state full-width.
       - Verify: map shows behind/above the pane; when the toggle is tapped again to collapse, map renders fully with no grey artifacts.

    J. Repeat collapsed↔list cycle three times on mobile emulation. Confirm no artifacts.

    Failure signals to report (any of these = NOT approved):
    - Grey untiled rectangles visible on the map at any point
    - Mapbox zoom controls or attribution overlapping the pane or floating outside their expected corner
    - Tiles visibly stretched or scaled (indicates the canvas size mismatch with container)
    - Console errors mentioning `resize`, `WebGL`, or `_paneState`
    - URL pane= param does not update on transition
    - Pane fails to open, close, expand, or shrink at any point

    If ANY failure signal appears: do NOT approve. Capture a screenshot and a 5-second screen recording, paste the failure description into the resume signal, and the next planner run will create a gap-closure plan adding an explicit `bee-map._map?.resize()` trigger (or fixing whatever root cause is identified).
