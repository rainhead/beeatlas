## Summary

Phase 16 adds `county` and `ecoregion_l3` columns to both `ecdysis.parquet` and `samples.parquet` using geopandas spatial joins against WA county and EPA Level III ecoregion boundaries. All core dependencies (geopandas 1.1.2, pyarrow, pyogrio) are already installed in the project. The ecoregion source file (`NA_CEC_Eco_Level3.zip`) is already present at `data/`. The county source requires downloading the 84 MB Census TIGER/Line national county file and filtering to Washington state (STATEFP='53').

The `within` predicate join leaves 408 specimens (~0.89%) with null ecoregion values (coastal/edge points outside polygon boundaries). A `sjoin_nearest` fallback eliminates all nulls. For `sjoin_nearest` accuracy, points and polygons must be reprojected to a projected CRS (EPSG:32610 — UTM zone 10N, appropriate for WA) before the nearest join; leaving both in EPSG:4326 triggers a geopandas warning about incorrect results.

A critical finding: the ecoregion GeoJSON at 0.005 degree simplification produces a 404 KB file, just over the 400 KB success criterion. The CONTEXT.md specifies 0.005 degrees as the simplification tolerance, but the planner must note this boundary condition. Using 0.006 degrees produces 357 KB. The planner should use 0.006 degrees to safely meet the 400 KB limit.

**Primary recommendation:** Implement `data/spatial.py` with `add_region_columns(df, counties_gdf, ecoregions_gdf)` interface; load boundaries in `build-data.sh` before pipelines run; use two-step sjoin pattern (within then sjoin_nearest on null rows with projected CRS).

### Pattern 1: Two-Step Spatial Join with Nearest Fallback

**What:** First join using `predicate='within'` (accurate); then `sjoin_nearest` with projected CRS for null rows.
**When to use:** Any point-in-polygon join where coastal/edge points may fall outside boundaries.

```python
# Source: verified by running against live data/ecdysis.parquet + NA_CEC_Eco_Level3.zip
import geopandas as gpd
import pandas as pd

def add_region_columns(
    df: pd.DataFrame,
    counties_gdf: gpd.GeoDataFrame,
    ecoregions_gdf: gpd.GeoDataFrame,
) -> pd.DataFrame:
    """Add county and ecoregion_l3 columns to df using spatial join.

    Both counties_gdf and ecoregions_gdf must already be in EPSG:4326.
    df must have 'longitude' and 'latitude' columns (ecdysis) or 'lon'/'lat' (iNat).
    """
    # Determine coordinate column names
    lon_col = 'longitude' if 'longitude' in df.columns else 'lon'
    lat_col = 'latitude' if 'latitude' in df.columns else 'lat'

    pts = gpd.GeoDataFrame(
        df,
        geometry=gpd.points_from_xy(df[lon_col], df[lat_col]),
        crs='EPSG:4326'
    )

    # Step 1: within join (accurate for EPSG:4326)
    joined = gpd.sjoin(pts, counties_gdf[['NAME', 'geometry']], how='left', predicate='within')
    joined = joined[~joined.index.duplicated(keep='first')]

    # Step 2: nearest fallback for null rows — project to avoid inaccuracy warning
    null_mask = joined['NAME'].isna()
    if null_mask.any():
        proj_crs = 'EPSG:32610'  # UTM zone 10N, appropriate for WA
        null_pts_proj = pts[null_mask].to_crs(proj_crs)
        counties_proj = counties_gdf[['NAME', 'geometry']].to_crs(proj_crs)
        nearest = gpd.sjoin_nearest(null_pts_proj, counties_proj, how='left')
        nearest = nearest[~nearest.index.duplicated(keep='first')]
        joined.loc[null_mask, 'NAME'] = nearest['NAME'].values

    df = df.copy()
    df['county'] = pd.array(joined['NAME'].values, dtype=pd.StringDtype())

    # Same two-step pattern for ecoregions
    joined_eco = gpd.sjoin(pts, ecoregions_gdf[['NA_L3NAME', 'geometry']], how='left', predicate='within')
    joined_eco = joined_eco[~joined_eco.index.duplicated(keep='first')]

    null_mask_eco = joined_eco['NA_L3NAME'].isna()
    if null_mask_eco.any():
        proj_crs = 'EPSG:32610'
        null_pts_proj = pts[null_mask_eco].to_crs(proj_crs)
        eco_proj = ecoregions_gdf[['NA_L3NAME', 'geometry']].to_crs(proj_crs)
        nearest_eco = gpd.sjoin_nearest(null_pts_proj, eco_proj, how='left')
        nearest_eco = nearest_eco[~nearest_eco.index.duplicated(keep='first')]
        joined_eco.loc[null_mask_eco, 'NA_L3NAME'] = nearest_eco['NA_L3NAME'].values

    df['ecoregion_l3'] = pd.array(joined_eco['NA_L3NAME'].values, dtype=pd.StringDtype())

    return df
```

### Pattern 2: GeoJSON Boundary Generation Script

**What:** Download-if-missing, filter to WA, simplify, write GeoJSON.
**When to use:** In `build-data.sh` before pipelines run.

```python
# Source: verified by running against live NA_CEC_Eco_Level3.zip
import geopandas as gpd
from pathlib import Path

DATA_DIR = Path(__file__).parent.parent  # data/
ASSETS_DIR = Path(__file__).parent.parent.parent / 'frontend' / 'src' / 'assets'

WA_BBOX = (-125.0, 45.5, -116.9, 49.1)  # minx, miny, maxx, maxy
ECO_ZIP = DATA_DIR / 'NA_CEC_Eco_Level3.zip'
TIGER_ZIP = DATA_DIR / 'tl_2024_us_county.zip'
TIGER_URL = 'https://www2.census.gov/geo/tiger/TIGER2024/COUNTY/tl_2024_us_county.zip'
ECO_URL = 'https://dmap-prod-oms-edc.s3.us-east-1.amazonaws.com/ORD/Ecoregions/cec_na/NA_CEC_Eco_Level3.zip'

SIMPLIFY_TOLERANCE = 0.006  # degrees; produces ~357 KB for ecoregions (safely under 400 KB)

def build_ecoregion_geojson():
    gdf = gpd.read_file(f'zip://{ECO_ZIP}!NA_CEC_Eco_Level3.shp')
    gdf = gdf.to_crs('EPSG:4326')
    gdf_wa = gdf.cx[WA_BBOX[0]:WA_BBOX[2], WA_BBOX[1]:WA_BBOX[3]]
    gdf_wa = gdf_wa[['NA_L3NAME', 'geometry']].copy()
    gdf_wa['geometry'] = gdf_wa.simplify(SIMPLIFY_TOLERANCE)
    out = ASSETS_DIR / 'epa_l3_ecoregions_wa.geojson'
    gdf_wa.to_file(out, driver='GeoJSON')

def build_county_geojson():
    gdf = gpd.read_file(f'zip://{TIGER_ZIP}')
    wa = gdf[gdf['STATEFP'] == '53'].copy()
    wa = wa.to_crs('EPSG:4326')
    wa = wa[['NAME', 'geometry']].copy()
    wa['geometry'] = wa.simplify(SIMPLIFY_TOLERANCE)
    out = ASSETS_DIR / 'wa_counties.geojson'
    wa.to_file(out, driver='GeoJSON')
```

### Pattern 4: iNat Pipeline Integration

**What:** Call `add_region_columns()` after final merge, before writing parquet.
**When to use:** In `inat/download.py:main()`.

The iNat pipeline currently uses pure pandas (no geopandas). Adding `import geopandas` is acceptable per locked decision. The spatial join must happen on the final merged DataFrame (after `merge_delta`), not on the delta, to ensure existing rows also get region columns on incremental runs.

**Key concern:** On incremental runs, existing rows in `samples.parquet` already written without `county`/`ecoregion_l3` will be re-read via `pd.read_parquet` which will fail if those columns don't exist yet. The first incremental run after this change reads the old parquet without region columns, merges, then the full merged DataFrame gets region columns added — this is correct because region columns are added to the merged output before writing.

However, if the existing parquet on disk was written by the old pipeline (no region columns), `merge_delta` will `pd.concat` a DataFrame with region columns (delta) against one without (existing). This creates NaN for region columns on existing rows. The spatial join on the merged result fills all rows — this works because `add_region_columns` rebuilds from lat/lon coordinates, not from existing column values.

### Anti-Patterns to Avoid

- **Joining on existing `county` DarwinCore field**: The Ecdysis CSV has a `county` field but it is sparse/unreliable — locked decision is to override with spatial join.
- **Leaving sjoin_nearest in EPSG:4326**: Produces a UserWarning "Geometry is in a geographic CRS. Results from 'sjoin_nearest' are likely incorrect." Must project to EPSG:32610 for nearest join.
- **Adding region columns to the iNat delta before merge**: Region columns on delta only would leave existing rows with NaN after merge. Add to final merged DataFrame.
- **Forgetting index deduplication after sjoin**: `gpd.sjoin` can produce multiple rows per input point when a point falls on a shared polygon boundary. Use `joined[~joined.index.duplicated(keep='first')]` defensively.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Point-in-polygon join | Custom loop with shapely.contains | `gpd.sjoin(predicate='within')` | Spatial index (STRtree) makes this orders of magnitude faster |
| Nearest polygon | Distance calculation loop | `gpd.sjoin_nearest` | Built-in, handles index and dedup |
| CRS reprojection | Manual coordinate math | `.to_crs('EPSG:4326')` | Handles datum transformations correctly |
| Shapefile reading | zipfile + custom parser | `gpd.read_file('zip://path!file.shp')` | Handles all shapefile formats, CRS reading |
| GeoJSON simplification | Custom Douglas-Peucker | `.simplify(tolerance)` | Well-tested, preserves topology option |

**Key insight:** geopandas abstracts spatial indexing and geometry operations that have subtle correctness requirements. Custom implementations inevitably miss edge cases (shared boundaries, polygon winding order, datum shifts).

## Common Pitfalls

### Pitfall 1: sjoin_nearest Geographic CRS Warning
**What goes wrong:** Calling `sjoin_nearest` on EPSG:4326 data produces `UserWarning: Geometry is in a geographic CRS. Results from 'sjoin_nearest' are likely incorrect.` Distance calculations on degrees are inaccurate near poles.
**Why it happens:** EPSG:4326 uses angular units (degrees); distance in degrees is not uniform.
**How to avoid:** Project both the null-points GDF and the boundary GDF to EPSG:32610 (UTM zone 10N) before `sjoin_nearest`. Verified: this eliminates the warning and produces 0 final nulls.
**Warning signs:** Yellow warning line in pipeline output containing "geographic CRS"

### Pitfall 2: Ecoregion File Size at 0.005 Degree Tolerance
**What goes wrong:** The CONTEXT.md specifies 0.005 degree simplification tolerance, but the generated ecoregion GeoJSON measures 404.2 KB — just over the 400 KB success criterion.
**Why it happens:** 80 WA ecoregion polygons with complex coastal geometry; 0.005 degrees is insufficient simplification.
**How to avoid:** Use 0.006 degrees tolerance (measured: 356.6 KB). This safely meets the 400 KB limit.
**Verification:** At task level, measure file size after generation: `ls -lh frontend/src/assets/epa_l3_ecoregions_wa.geojson`

### Pitfall 3: iNat Incremental Run Column Mismatch
**What goes wrong:** First incremental run after this phase reads existing `samples.parquet` (no region columns), `merge_delta` creates NaN region columns on existing rows.
**Why it happens:** `pd.concat` of two DataFrames with different columns fills missing with NaN.
**How to avoid:** Apply `add_region_columns()` to the full merged DataFrame (after `merge_delta`), not just to the delta. This rebuilds region assignments for all rows from coordinates.
**Warning signs:** Region columns present but with NaN values in older rows

### Pitfall 4: Ecoregion CRS Not Reprojected Before sjoin
**What goes wrong:** Using `gpd.sjoin` on the ecoregion GDF while it is still in the Lambert AEA projection (non-EPSG spherical) produces silently wrong results — points miss polygons they should fall within.
**Why it happens:** Lambert AEA uses meters; specimen coordinates are in degrees; the join fails to match.
**How to avoid:** Always call `eco_gdf.to_crs('EPSG:4326')` immediately after loading from the zip. The `spatial.py` module should accept pre-reprojected GDFs (enforced by convention) OR reproject internally.
**Warning signs:** Abnormally high null count after `within` join (would be thousands, not 408)

### Pitfall 5: sjoin Duplicate Rows on Shared Boundaries
**What goes wrong:** A point that falls exactly on a shared polygon boundary may match both polygons, producing two output rows per input row. Downstream processing assumes 1:1 row correspondence.
**Why it happens:** `sjoin` with `predicate='within'` on boundary points can match both adjacent polygons.
**How to avoid:** Always apply `joined[~joined.index.duplicated(keep='first')]` after any `sjoin` call.
**Warning signs:** Output DataFrame has more rows than input
