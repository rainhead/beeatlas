<objective>
Establish the `taxon_id` contract layer in `src/filter.ts` that every Phase 130
consumer builds against: change `FilterState` / `TaxonOption` / `FilterChangedEvent`
to key on integer `taxonId`, rewrite the `buildFilterSQL` taxon clause to a
materialized-path descendant query against the shipped `taxa` table, and add
`taxon_id` to `OCCURRENCE_COLUMNS` / `OccurrenceRow`. Also update the `emptyFilter()`
test helpers in both affected test files so the suite keeps compiling, and add the
descendant-clause unit tests.

This is the interface-first / Wave 0 plan: it defines the contracts (types + SQL
shape) and the failing-then-passing tests that the autocomplete (Plan 02) and
detail-card (Plan 03) plans consume. It is additive — the `family`/`genus`/
`scientificName` string columns stay present in the DB and are simply no longer used
for taxon matching.

Purpose: Close MFILT-01 (descendant `taxon_id` filtering) at the query layer and
unblock the consumer plans with a stable typed contract.
Output: `src/filter.ts` on the new contract; descendant-clause tests green; both
test helpers updated.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-130-01 | Tampering / Injection | `buildFilterSQL` taxon clause | mitigate | `taxonId` is a TypeScript `number`; interpolate it as a bare integer (`'/${f.taxonId}/'` and `taxon_id = ${f.taxonId}`). No string value reaches the WHERE clause, so no `''`-escaping is needed and no untrusted string is interpolated. Upstream (Plan 02) guarantees `taxonId` is set only from `parseInt`-validated input or a cache-resolved integer; this plan's contract types it as `number | null`. |
| T-130-02 | Information disclosure | descendant subquery against `taxa` | accept | Read-only bundled DB; `taxa`/`occurrences` are public reference data. No PII; no sensitive rows. |
