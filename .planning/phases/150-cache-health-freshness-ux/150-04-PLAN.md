<objective>
Land the user-facing surfaces for Phase 150: the `<bee-header>` ready-pill (CACHE-01), determinate-progress inline bar (CACHE-02), freshness caption + popover (CACHE-04), storage-estimate row (CACHE-03), and the `<bee-atlas>` non-modal SW-update banner (OFF-03 carry).

`<bee-atlas>` is the state owner. It gains 5 new `@state` fields (`_cacheState`, `_primeProgress`, `_updateAvailable`, `_freshnessLabel`, `_storageEstimate`), 3 new window listeners (`cache-prime-progress`, `cache-state-changed`, `sw-update-available`), 1 element listener (`cache-popover-toggle` from `<bee-header>`), and 1 new render branch (the bottom `.update-banner`). All 5 new states are relayed to `<bee-header>` via `@property`.

`<bee-header>` is the pure presenter. It gains 5 new `@property` receives + 1 internal `_popoverOpen` `@state`, the `.ready-pill` chrome (reusing the `.offline-pill` base style), the `.freshness-caption` sub-line under `<h1>BeeAtlas`, and the `.cache-popover` dialog. It emits `cache-popover-toggle` upward so `<bee-atlas>` can lazy-fire `navigator.storage.estimate()` only when the popover opens (D-18).

Per `<phase_specifics>`: both Lit element files move in one atomic plan because the `<bee-header>` `@property`s are owned by `<bee-atlas>` — splitting them would leave an intermediate state with a state-owner/presenter contract mismatch.

Wave 0 test coverage in `src/tests/cache-state.test.ts` (new file per orchestrator's quality_gate) pins every UI-SPEC pill state, the popover visibility rules, the update-banner lifecycle, and the lazy storage-estimate path.

Output: A user-tested ready surface — fresh visit → pill says "Caching… N%" with an inline determinate bar; full prime → pill says "✓ Offline-ready"; pill tap → popover with freshness + storage details; new SW → bottom banner with tap-to-reload + session-dismiss.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| page script → window CustomEvents (`cache-prime-progress`, `cache-state-changed`, `sw-update-available`, `cache-update-acted`) | A same-origin script can forge any of these CustomEvents to manipulate the `<bee-atlas>` state. The threat is bounded — see T-150-01 below. |
| `<bee-header>` shadow root → popover dismiss handlers (`document.addEventListener('click')`) | A page-level click handler could pre-empt the click and prevent popover dismiss. Mitigated by using `composedPath()` checks inside the handler so legitimate ready-pill clicks don't accidentally close the popover. |
| `<bee-atlas>._onBannerTap` → `window.__wb.messageSkipWaiting()` + `window.location.reload()` | Both are gated on user click; same-origin script could synthesize the click to trigger an early reload — bounded by no data loss (Phase 149 D-04 + atomic cache.put). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-150-01 (carry from Plan 02) | Spoofing | `window.addEventListener('sw-update-available', this._onSwUpdateAvailable)` in `<bee-atlas>` | mitigate | A same-origin script can dispatch a fake event and force the banner to render. The banner's tap-handler then calls `window.__wb?.messageSkipWaiting()` — but per RESEARCH §Pattern 2, workbox-window's `messageSkipWaiting()` is a no-op when there is no waiting SW (it checks `registration.waiting != null` internally). Worst case is a confusing UI moment, not an unauthorized cache state change. Severity: low. |
| T-150-02 (carry from `<phase_specifics>`) | Tampering | `<bee-header>` ready-pill progress fill width derived from `received / total` | mitigate | The pill renders `pct = Math.max(0, Math.min(99, Math.floor(received / total * 100)))` — two-sided clamp prevents both overflow (`total === 0` would give Infinity → clamped to 99) and underflow (negative `received` → clamped to 0). The orchestrator's `total` is additionally validated as a positive finite number in Plan 03's `primeAll`. Two-layer defense. Severity: low. |
| T-150-05 | Information Disclosure | `<bee-header>` popover's storage estimate row | accept | The displayed `usageMB` and `quotaMB` numbers are derived from `navigator.storage.estimate()` and reflect user data the user already owns. The threat surface is solely visual — there is no exfiltration path. Severity: informational. |
| T-150-06 | Spoofing | Popover passive update affordance (Row 4 — fires `cache-update-acted`) | mitigate | Same surface as the banner tap. Both routes call `<bee-atlas>._onBannerTap`, which gates on `window.__wb?.messageSkipWaiting()` (no-op when no SW waiting). The popover row is only rendered when `updateAvailable === true`, and a forged `sw-update-available` event would have to land first — already classified as T-150-01. Severity: low. |
