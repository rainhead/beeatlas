<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| page script → `window.__wb` global | Any same-origin script can read `window.__wb` and call `wb.messageSkipWaiting()` after Plan 02 lands. The threat surface is bounded by the SW-side `SKIP_WAITING` message gate from Plan 01 — same threat class as T-150-01 in Plan 01 (a malicious page script can advance the SW update cycle by one step but cannot corrupt cached data). |
| page → window 'sw-update-available' CustomEvent | Plan 02 dispatches this event. Plan 04's `<bee-atlas>` listener will mount a banner on receipt. A same-origin malicious script could forge the CustomEvent to show a spoofed banner; the banner tap-handler then calls `window.__wb?.messageSkipWaiting()` — which is harmless if no real SW is waiting (workbox-window verified its `messageSkipWaiting()` is a no-op when `registration.waiting` is null per RESEARCH §Pattern 2). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-150-01 (carry from Plan 01) | Spoofing | `window.dispatchEvent(new CustomEvent('sw-update-available'))` | mitigate | The forged-event surface is constrained: a same-origin script can show a fake banner but cannot fake the SW activation. Plan 04 will note that the banner only invokes `wb.messageSkipWaiting()` which workbox-window guards on `registration.waiting != null`. Severity: low. |
| T-150-02 | Tampering | `window.__wb` global | accept | `window.__wb` is a runtime convenience for the cross-module handoff to the banner tap-handler. Replacement of the global by a same-origin script would short-circuit the update prompt but not corrupt cache state. The alternative (a module-level singleton with named export) breaks the no-SW-on-/ structural guarantee from 147 — exports could be imported by `_pages/index.html`'s entry. The `window.__wb` shape is the lesser evil. Severity: low. |
</threat_model>
