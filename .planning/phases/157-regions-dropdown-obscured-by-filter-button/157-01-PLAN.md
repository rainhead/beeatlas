<objective>
Fix the regions dropdown being obscured by the filter button by relocating the region control out of `<bee-map>`'s shadow DOM into `<bee-atlas>`, where it becomes a sibling of `<bee-pane>` in `.content` and can be stacked above it — WITHOUT removing `bee-map`'s load-bearing `z-index: 0`. Also lay the collapsed filter button beside (not below) the regions button.

Purpose: The region menu lives inside `<bee-map>`, which is `z-index: 0` in `.content` — a stacking context that traps the menu below the sibling `<bee-pane>` (`z-index: 1`). Raising the menu's local z-index cannot escape that context. Removing `bee-map`'s `z-index: 0` is rejected: it was added deliberately (commit 014c6d15 / Phase 108-02) so Mapbox controls — notably the bottom-right attribution that overlaps the table pane — cannot bleed above `<bee-pane>`. The robust fix (operator-approved) lifts ONLY the region control out of that context by moving it to `<bee-atlas>`.

Output:
- `src/bee-atlas.ts`: the region control (markup + CSS + `_regionMenuOpen` state + `_toggleRegionMenu`/`_selectBoundaryMode` + `_onDocumentClick` outside-close) relocated here as a sibling of `<bee-pane>`, with `.region-control` z-index above `<bee-pane>`; selection routed through a shared `_applyBoundaryMode()` extracted from `_onBoundaryModeChanged`; the collapsed `<bee-pane>` laid out beside the regions button; `bee-map { z-index: 0 }` retained.
- `src/bee-map.ts`: region control removed; `boundaryMode` @property and map-click boundary logic unchanged.
- `src/tests/bee-atlas.test.ts`: `STACK-01` source-analysis describe block.
- `src/tests/bee-map.test.ts`: any region-control assertion updated to reflect the relocation (none currently expected).
</objective>
