## Overview

Rehouse the existing Vite SPA inside an Eleventy build pipeline without changing URLs or runtime behavior, establishing the static-site-generator foundation needed for content pages (Species tab, welcome page) in v3.2 and beyond. Two phases: install Eleventy + `@11ty/eleventy-plugin-vite` and hoist `frontend/` to the repo root (Phase 74), then lay down the empty Eleventy authoring scaffold — two-layer Nunjucks layout chain with embedded `<bee-header>` chrome — proven by an orphan diagnostic page at `/_scaffold-check/` (Phase 75).

**Out of scope (deferred to v3.2 Species Tab):**
- Moving the SPA to `/collection`
- Welcome/about page at `/`
- Lit SSR decision for new Eleventy pages
- Any new content pages

**Pattern reference:** `~/dev/pnwmoths` ships an analogous Eleventy + Vite static site (`@11ty/eleventy-plugin-vite`).

## Milestone Summary

**Decimal Phases:** None.

**Key Decisions:**

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Single-package layout (collapse `frontend/` to repo root) | Workspaces add CI/script complexity for a one-package repo; root-level `package.json` is simpler for a static site | ✓ Good — Phase 74 atomic Layout B move; CI scripts dropped `--workspace=frontend` cleanly |
| `index.html` must be templated, not passthrough | `@11ty/eleventy-plugin-vite` short-circuits when `results.length === 0`; passthrough `index.html` does not register as an Eleventy result | ✓ Good — Phase 74 deviation; `_pages/index.html` ships as Liquid no-op |
| Two-step publicDir pipeline (Eleventy passthrough → Vite publicDir) | Single-owner publicDir is wrong for the rename-and-rm mechanics of the Eleventy + Vite plugin | ✓ Good — Phase 74; `public/` → `_site/public/` → `.11ty-vite/public/` → final `_site/` |
| Plugin `viteOptions` pass-through for dev (`envDir`, `optimizeDeps.exclude`) | Build pass reads `vite.config.ts`; dev pass via Eleventy plugin does NOT — must set via `viteOptions` | ✓ Good — Phase 74 fix `5bfca85`; HMR + WASM resolution work in dev |
| Orphan diagnostic page at `/_scaffold-check/` (permanent fixture) | Deploy diagnostic for the multi-entry Vite pipeline; permanent so any production breakage of layout chain or bee-header bundling shows up there | ✓ Good — Phase 75 D-01; one-time research uncertainty resolved |
| Nunjucks templating (over Liquid for new pages) | Existing pnwmoths pattern; Eleventy `.njk` extension is the convention; SPA `index.html` continues as Liquid no-op | ✓ Good — Phase 75 D-02 |
| Two-layer layout chain via Eleventy front-matter `layout:` + `{{ content \| safe }}` | NOT Nunjucks `{% extends %}`/`{% block %}` — mixing the two breaks rendering. Pnwmoths-pattern verbatim | ✓ Good — Phase 75 D-03 (Pitfall #1) |
| `<bee-header>` Lit component embedded in `default.njk`; chrome renders via side-effect Vite entry | One-line `import` modules in `src/entries/` whose sole purpose is to be Rollup entries that trigger custom-element registration via `@customElement` decorators | ✓ Good — Phase 75 D-04; pattern reusable for v3.2 species pages |
| Multi-entry Vite build via plugin's HTML processor (`appType: "mpa"`) | Vite walks every emitted templated HTML page and produces a hashed bundle automatically — no `rollupOptions.input` config needed; no manifest required | ✓ Good — Phase 75 D-05; load-bearing assumption A5 verified end-to-end |
| `dir.{includes,layouts,data}` resolve relative to `dir.input` in Eleventy 3.x | When `dir.input` is a leaf folder (e.g. `_pages/`) but layout files live at repo root, use `"../_<name>"` traversal | ✓ Good — Phase 75 Rule 1 deviation; documented inline in `eleventy.config.js` |
| Underscore-prefix-in-URL via `permalink:` (not filename) | Eleventy treats leading underscore in input filenames ambiguously; using `permalink: /_scaffold-check/index.html` with non-underscore filename sidesteps the ambiguity | ✓ Good — Phase 75 Pitfall #2 |
| `_data/build.js` default-export-an-object pattern | Build-time metadata feed; v3.2 can add `_data/species.js`, `_data/photos.js` etc. following the same shape | ✓ Good — Phase 75; pattern established for v3.2 |
