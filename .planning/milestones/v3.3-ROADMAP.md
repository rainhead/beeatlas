## Overview

A time-boxed exploratory spike to learn whether `dbt-duckdb` is the right shape for the BeeAtlas data layer. One representative slice of `export.py` (→ `occurrences.parquet` + `counties.geojson` + `ecoregions.geojson`) was ported end-to-end on a branch, exercising dbt's test surface, model contracts, partial-run behavior, and lineage. The deliverable is `.planning/research/dbt-spike-findings.md` with a concrete go/no-go recommendation and a 5-prerequisite checklist for a follow-up rewrite milestone. No production paths were modified.

## Milestone Summary

**Key Accomplishments:**

1. **dbt-duckdb project scaffolded** at `data/dbt/` — 23 models (11 staging + 9 intermediate + 3 marts) plus the `emit_feature_collection` Jinja macro, with three-layer materialization and a `materialized='external'` mart producing `occurrences.parquet` (47,883 rows, 1.2 MB) byte-comparable to `export.py`'s output.
2. **Test surface exercised across three dimensions** — 10 dbt generic tests (8 PASS, 1 FAIL, 1 ERROR — both awkward-fit findings documented); a 33-column enforced contract on `occurrences` with verified compile-time drift detection; 11 pytest diff functions covering row count, schema (names+types), `ecdysis_id` + `host_observation_id` key-set parity, spatial-join diffs, and GeoJSON parity.
3. **Material diffs root-caused** — only divergences from `export.py` are: 84 county-boundary rows (ST_Within nondeterminism in *both* implementations — semantic divergence to investigate) and GeoJSON whitespace formatting (neutral/cosmetic). Zero divergence on row count, schema, key sets, ecoregion_l3, GeoJSON features, or property names.
4. **Partial-run + lineage characterized** — `dbt build --select` exercised on two subgraphs (`staging+` 23 models; `+occurrences` 21 models); DuckDB serialization-despite-scheduler-parallelism behavior pinned down; 23-model lineage artifact captured.
5. **Concrete verdict + prerequisites** — `.planning/research/dbt-spike-findings.md` ends with `GO-WITH-CONDITIONS` and a 5-prerequisite checklist (test surface, samples.parquet schema decision, ingestion-vs-transform boundary, parallel-run/orchestration story, frontend impact) that becomes v3.4 input.
6. **Pinned dbt-core==1.10.1** after discovering a 1.10.20 regression (`KeyError: 'javascript'`); recorded in commit `fffc496` so future contributors don't hit the same wall.

**Key Decisions:**

- Single slice, end-to-end (no multi-slice porting) — one slice is enough to learn the shape.
- Mart `occurrences` retains the FULL OUTER JOIN single-output shape from `export.py` (samples are not materialized as a separate `samples.parquet`), preserving production parity even though `REQUIREMENTS.md` originally named `samples.parquet` separately. Reconciliation deferred to v3.4+.
- GeoJSON emission uses `COPY ... FORMAT CSV, DELIMITER '', QUOTE '', HEADER false` with explicit `::VARCHAR` cast as a workaround for DuckDB `FORMAT JSON` envelope wrapping. Recorded as a deviation and a v3.4 prerequisite to clean up.
- `materialized='external'` for the `occurrences` mart confirmed compatible with `contract: enforced: true` in dbt-duckdb 1.10.1 — the contract fires at compile-time before any parquet write.
