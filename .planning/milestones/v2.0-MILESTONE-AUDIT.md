## Integration Check Results

### ✓ PASS — Core wiring complete

- `bee-atlas` correctly shows `<bee-table>` when `viewMode=table`, `<bee-map>` when `viewMode=map`
- `bee-table` receives all required props: `rows`, `rowCount`, `layerMode`, `page`, `loading`
- Filter changes trigger `_runTableQuery` → `<bee-table>` updates
- Layer mode changes trigger `_runTableQuery` → column sets switch
- `download-csv` event wired: `bee-table` → `bee-atlas._onDownloadCsv` → `queryAllFiltered` → blob download
- URL round-trip for viewMode: encode/decode verified in url-state.ts

### ⚠ GAP 1 — Direct-URL table view shows empty table (functional bug)

**File:** `bee-atlas.ts`, `firstUpdated` (around line 257)
**Issue:** When the user navigates directly to `?view=table`, `_loadSummaryFromDuckDB` is called after DuckDB is ready, but `_runTableQuery` is never called. The `data-loaded` event that normally triggers the first table query comes from `<bee-map>`, which is not rendered in table mode.
**Result:** Table loads with 0 rows. User must switch to map and back to populate.
**Fix:** Add `if (this._viewMode === 'table') this._runTableQuery()` at the end of the DuckDB-ready callback in `firstUpdated`.
**Severity:** Medium — broken UX for bookmarked table URLs.

### ⚠ GAP 2 — Sort-by-column removed, not officially deferred

**Issue:** TABLE-04 (sort by column header) was implemented in Phase 40 plan 40-01, then removed in a post-phase UI refactor (commit `8fbe895`). The refactor was intentional but TABLE-04 was not updated in REQUIREMENTS.md.
**Sort is now hardcoded:** specimens by `date DESC, recordedBy ASC, fieldNumber ASC`; samples by `date DESC, observer ASC, sample_id ASC`.
**Action needed:** Either re-implement sort or formally mark TABLE-04 as deferred in requirements.
**Severity:** Low — data is navigable without column sort; hardcoded order is reasonable.

### ℹ INFO — `_tableLoading` spinner absent on direct URL table load

Related to Gap 1. No loading state shown during DuckDB init when starting in table view. Cosmetic.
