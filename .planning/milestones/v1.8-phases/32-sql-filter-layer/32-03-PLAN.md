<task type="auto">
  <name>Task 1: Eagerly load county and ecoregion sources on init</name>
  <files>frontend/src/region-layer.ts, frontend/src/bee-map.ts</files>
  <action>
Gap: countySource and ecoregionSource are OL VectorSources with `url` config. OL lazy-fetches these only when the source is used by a visible layer. Since regionLayer starts `visible: false`, the GeoJSON never loads, and the `countySource.once('change')` handler in bee-map.ts never fires.

Fix in region-layer.ts: After creating countySource and ecoregionSource, call `.loadFeatures()` on each to trigger an eager fetch. OL VectorSource.loadFeatures() triggers the URL fetch regardless of layer visibility. Call it with the extent of the world in EPSG:3857, the resolution 1, and the source's projection:

```typescript
import { get as getProjection } from 'ol/proj.js';
const worldExtent = getProjection('EPSG:3857')!.getExtent()!;
countySource.loadFeatures(worldExtent, 1, getProjection('EPSG:3857')!);
ecoregionSource.loadFeatures(worldExtent, 1, getProjection('EPSG:3857')!);
```

Add these two calls at module scope, right after the source declarations (before the regionLayer declaration). This ensures the `once('change')` handlers in bee-map.ts will fire on page load.

No changes needed to bee-map.ts for this task — the existing `countySource.once('change')` and `ecoregionSource.once('change')` handlers (lines 822-831) will now fire as expected since the sources will load eagerly.

<task type="auto">
  <name>Task 2: Fix _applyFilter to preserve county/ecoregion selections through _setBoundaryMode</name>
  <files>frontend/src/bee-map.ts</files>
  <action>
Gap: In `_applyFilter` (line 567), filterState.selectedCounties and selectedEcoregions are set (lines 574-575), then `_setBoundaryMode` is called (line 577) which clears them (lines 273-274) when mode is 'off'. The subsequent `_runFilterQuery()` (line 581) sees empty sets, `isFilterActive` returns false for the county/ecoregion part, `queryVisibleIds` returns nulls, and `filteredSummary` is set to null — sidebar shows unfiltered totals.

Fix: Modify `_setBoundaryMode` to accept an optional parameter that skips clearing the filterState selections and skips the internal `_runFilterQuery()`. When called from `_applyFilter`, pass this flag since `_applyFilter` manages filterState and runs its own query afterward.

Specifically:
1. Change `_setBoundaryMode` signature to: `private async _setBoundaryMode(mode: 'off' | 'counties' | 'ecoregions', skipFilterReset = false): Promise<void>`
2. In the `mode === 'off'` branch, wrap the two `filterState.selected*` clear lines in `if (!skipFilterReset)`:
   ```typescript
   if (mode === 'off') {
     regionLayer.setVisible(false);
     if (!skipFilterReset) {
       filterState.selectedCounties = new Set();
       filterState.selectedEcoregions = new Set();
     }
   }
   ```
3. Also wrap the `await this._runFilterQuery()` call (line 282) in `if (!skipFilterReset)`:
   ```typescript
   if (!skipFilterReset) {
     await this._runFilterQuery();
   }
   ```
4. In `_applyFilter` (line 577), pass `true` as the second argument:
   ```typescript
   await this._setBoundaryMode(detail.boundaryMode, true);
   ```

This way: when _applyFilter calls _setBoundaryMode, the county/ecoregion selections are preserved in filterState, and the redundant _runFilterQuery inside _setBoundaryMode is skipped. The _runFilterQuery on line 581 of _applyFilter then runs with the correct filterState, producing correct visibleEcdysisIds, and the filteredSummary computation (lines 591-605) correctly reflects filtered totals.

Do NOT change any other call sites of _setBoundaryMode (polygon click handler, boundary tab switching) — those should continue to clear filterState as before.
