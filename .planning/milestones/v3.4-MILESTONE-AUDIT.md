## 5. Tech Debt (INFO findings)

These are non-blocking observations from the integration checker. All three are already tracked or have a clear next step.

### From phase 088
1. **VALIDATE-01 retirement decision not recorded.** `data/tests/test_dbt_diff.py` was the cross-phase regression gate during ports. Post-cutover both sides of every assertion are dbt-produced (sandbox vs. public, where public was just copied from sandbox by `_run_dbt_build`). REQUIREMENTS.md anticipated this becomes vacuous and asked for retirement-or-repurpose. The harness is still in the tree, still passing, but no decision has been recorded. **Recommend:** add a short ADR or todo deciding whether to retire (delete tests) or repurpose (compare against a frozen snapshot for regression detection).

2. **Stale comment-only references to validate-schema.mjs.** Two sibling scripts (`scripts/validate-species.mjs:9`, `scripts/validate-bundle-size.mjs:9`) contain historical "mirrors this idiom of" prose mentioning validate-schema.mjs. Not executable; no functional impact. Cleanup is a 2-minute quick task whenever the user is in that file.

### From phase 086
3. **species.parquet split read/write path.** `_run_dbt_build` does NOT copy `species.parquet` from sandbox to `EXPORT_DIR`. Instead, `species_export.py` reads from `DBT_SANDBOX_DIR` and writes the 19-col version (sandbox + slug) directly to `EXPORT_DIR`. This is documented (run.py:62–64, species_export.py:37–44), but it does mean that if `species_export.py` were skipped or failed silently, the published `species.parquet` would be stale relative to a freshly-built sandbox. `run.py` STEPS re-raises on exception so this is guarded — flagging only as architecture context for future maintainers.
