# Milestone v3.7: Places

**Status:** ✅ SHIPPED 2026-05-18
**Phases:** 97–100.1 (including INSERTED Phase 100.1)
**Total Plans:** 11

## Overview

Add a curated directory of collecting locations (parks, reserves) with land owner info, permit records, specimen counts, and full map and filter integration. Hand-curated TOML data model; pipeline spatial join; static Eleventy pages; Mapbox Places boundary mode.

## Phases

### Phase 97: Place Data Model

**Goal**: The coordinator can define curated collecting locations in a TOML file that the build validates for correctness before the pipeline runs
**Depends on**: Phase 96
**Requirements**: PLC-01, PLC-02, PLC-03, PLC-04
**Plans**: 2 plans

Plans:

- [x] 097-01-PLAN.md — Create content/places.toml seed entries and data/places_validation.py validation module
- [x] 097-02-PLAN.md — Wire validation step into run.py STEPS and write pytest tests

**Details:**

Success criteria:
1. A coordinator can add an entry to `content/places.toml` with slug, name, land_owner, geometry_wkt (WGS84), and a permits array; the build accepts it
2. Each permit record carries issuing_authority, optional permit_number, nullable expiry_date, and type (project-level vs site-level)
3. The build fails with a descriptive error if any place has an invalid geometry, non-WGS84 CRS, duplicate slug, or slug characters outside `[a-z0-9-]`
4. The build fails if any two place polygons overlap (ST_Intersects check)
5. A pytest fixture with one valid and one invalid place entry verifies the pass/fail boundary

Completed: 2026-05-18

## Milestone Summary

**Key Decisions:**

- `land_owner` field name (not `owner`) per PLC-01 — avoids ambiguity between organizational and legal ownership
- `LOAD spatial` only in places_validation.py (not `INSTALL spatial`) — spatial extension already installed in pipeline DuckDB environment
- `promoteId: 'slug'` for places GeoJSON source in Mapbox — stable feature IDs across source reloads (not `generateId: true`)
- Two export artifacts: `places.geojson` (slim: slug + geometry, for Mapbox) and `places.json` (rich: all metadata + counts, no geometry, for Eleventy)
- D-01 (Phase 99): Permit display removed from v3.7 place pages — static hosting + legal sensitivity of permit data; revisit in v3.8+
- `leavingPlaces` conditional in `_onBoundaryModeChanged` avoids redundant filter query when switching between non-places modes
- `placeImplied` logic in `parseParams`: if `place=` present and no explicit `bm=`, derive `bm=places` — ensures URL deep-links from place pages set the correct boundary mode

**Issues Resolved:**

- B-01: place-maps/*.svg not uploaded to S3/CDN — fixed in Phase 100.1 (commit c7d7a31); was latent (both seed polygons had 0 occurrences at audit time, but would have 404'd in production once data populates)
- W-01: `_onBoundaryModeChanged` did not clear `selectedPlace` on mode switch — fixed in Phase 100.1 (commit 1ce1e40)

**Issues Deferred:**

- Phase 98 VERIFICATION.md missing — procedural gap; all code verified via SUMMARY files and code inspection
- W-02: `places_validation.py` validates slug/geometry/overlap but does NOT enforce PLC-02 required permit fields at runtime (issuing_authority, type) — malformed permit record loads silently
- W-03: `run.py` module docstring lists stale pipeline order omitting places-load, places-export, places-maps, topology-postprocess
- Nyquist compliance gaps: phases 97, 98, 100 missing VALIDATION.md; Phase 98 Wave 0 RED tests never written as stubs

**Technical Debt Incurred:**
