<objective>
Wire the `place-selected` event from `bee-map` and the extended `FilterChangedEvent` from `bee-filter-panel` into `bee-atlas` so that selecting a place on the map (or removing the chip in the panel) drives `_filterState.selectedPlace`, triggers `_runFilterQuery`, and pushes `place=<slug>` into the URL via the existing `_pushUrlState` mechanism. Extend the three `_filterState`-construction sites in `bee-atlas.ts` (init, `_init`, popstate) with `selectedPlace`. Widen `_boundaryMode` and `_onBoundaryModeChanged` to accept `'places'`. Add a Vitest integration test that drives the event from `bee-map` → bee-atlas and asserts the resulting state.

Purpose: This plan closes PMAP-02 (click polygon applies filter — Plan 02 emitted the event; this plan handles it) and PMAP-04 (URL deep-link round-trip — Plan 01 implemented encoding/parsing; this plan applies the parsed state and emits new state on user action). After this plan, the user can: open `/?place=ebeys-landing` and see the place pre-filtered with the polygon highlighted; click another polygon and watch the URL update; click the same polygon to deselect; remove the chip and watch both the URL and the highlight clear.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CustomEvent → component state | `place-selected` event's `detail.slug` is copied into `_filterState.selectedPlace` |
| URL → component state | `window.location.search` parsed by url-state.parseParams flows into `_filterState` and `_boundaryMode` |
| popstate event → component state | Browser back/forward triggers re-parse and re-apply |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-100-08 | Tampering | URL parameter `?place=<arbitrary>` → bee-atlas _filterState | mitigate | Validation happens downstream: buildFilterSQL escapes the value (Plan 01); the SQL clause produces zero rows for any slug not in the data set. The frontend treats the slug as an opaque string. No reflected XSS surface: the value is stored in state and rendered through Lit text interpolation (Plan 02 chip), never as HTML. URL of arbitrary length is handled by the existing URLSearchParams API — no buffer overflow surface in JavaScript. |
| T-100-09 | Repudiation | URL state updates via `_pushUrlState` from `_onPlaceSelected` | accept | URL history is client-only and not authoritative. `pushState` is the same mechanism used by all existing filter handlers (county, ecoregion, taxon). No new repudiation surface. |
| T-100-10 | Information disclosure | A shared link `/?place=<sensitive-slug>` could disclose a place filter | accept | Place metadata is intentionally public (Phase 99 ships place static pages with the same slug-based URLs). No privileged or PII surface in scope. |
| T-100-SC | Tampering (supply chain) | npm/pip/cargo installs | mitigate | No new package installs in this plan. All work extends existing source files. No package legitimacy audit required. |
</threat_model>
