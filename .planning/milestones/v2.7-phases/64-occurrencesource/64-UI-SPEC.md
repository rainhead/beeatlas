## Cluster Dot Sizing Contract

**This is the primary visual deliverable of Phase 64.**

The current radius formula in `style.ts` line 99:

```
radius = displayCount <= 1 ? 4 : 6 + Math.log2(Math.max(displayCount, 1)) * 2
```

For `displayCount = 1`, radius = 4px → dot diameter = 8px. This is below the 44px mobile tap target minimum (CONTEXT.md D-02).

**Required formula adjustment:**

```
minRadius = 22  // enforces 44px touch target
radius = Math.max(minRadius, displayCount <= 1 ? 22 : 22 + Math.log2(Math.max(displayCount, 1)) * 2)
```

Wait — the 44px rule applies to interactive tap targets; a single-dot with only 1 occurrence should still meet this. However the existing behavior of small dots (radius 4–6) is intentional visual density. CONTEXT.md D-02 says "Minimum rendered cluster dot diameter ≥ 44px" — apply this as:

- **Single-feature dot:** radius = 22 (diameter 44px) — minimum tap target
- **Multi-feature cluster, count 2–9:** radius = 22 (diameter 44px minimum; increase if formula gives larger)
- **Multi-feature cluster, count ≥ 10:** radius = max(22, 6 + log2(count) * 3) — scale up from baseline

Revised formula:
```ts
const BASE_RADIUS = 22;  // 44px diameter — minimum mobile tap target
const radius = displayCount <= 1
  ? BASE_RADIUS
  : Math.max(BASE_RADIUS, 6 + Math.log2(Math.max(displayCount, 1)) * 3);
```

Text label: use `bold 11px sans-serif` (unchanged); omit label when `displayCount === 1` (dot too small to need count).

## Interaction Contract

### Cluster Click → URL Encoding

**Single-feature click:**
- URL param: `o=ecdysis:1234` (specimen-backed) or `o=inat:5678` (sample-only)
- Format unchanged from prior phases; `parseParams` extended to accept `inat:` prefix
- Source: CONTEXT.md D-05

**Multi-feature cluster click:**
- URL param: `o=@lon,lat,radiusM`
  - `lon`, `lat` = mean WGS84 centroid of all features in cluster, 4 decimal places
  - `radiusM` = haversine distance in metres from centroid to furthest feature, rounded up to nearest integer metre
  - Example: `o=@-120.5123,47.4567,312`
- Source: CONTEXT.md D-06

**URL restore for `@lon,lat,r`:**
- Spatial query against `occurrences` table using equirectangular approximation (wa-sqlite has no trig; flat-earth acceptable for WA state extents and small cluster radii)
- Returns both specimen-backed and sample-only features within radius
- Source: CONTEXT.md D-07

### SelectionState Discriminated Union

```ts
type SelectionState =
  | { type: 'ids'; ids: string[] }
  | { type: 'cluster'; lon: number; lat: number; radiusM: number };
```

`buildParams` and `parseParams` handle both variants. `bee-atlas.ts` `_restoreSelectionSamples` handles both in the restore path.
Source: CONTEXT.md D-08

### layerMode (no visible effect)

`layerMode` property remains on `bee-map.ts` and `bee-atlas.ts` — no UI visibility change. The layer-switching toggle is NOT removed in this phase (Phase 65 concern). The property is a no-op because there is now only one source/layer.
Source: CONTEXT.md D-09

## Phase-Specific Notes

1. **No new DOM components.** Phase 64 adds `OccurrenceSource` (a TypeScript class extending `VectorSource`) and updates URL state types. No new HTML elements, no new CSS classes, no new user-visible UI surfaces.

2. **Canvas rendering only.** The visual change is exclusively on the OL map canvas — cluster dots rendered via `makeClusterStyleFn`. The dot sizing adjustment (44px minimum tap target) is the sole visual delta.

3. **Mixed-type cluster sidebar display is not polished.** Clicking a cluster containing both specimen-backed and sample-only features will display whatever `buildSamples()` in `bee-map.ts` produces from non-null columns. Null species data on sample-only features results in blank species entries. This is acceptable for Phase 64; Phase 65 replaces the detail component entirely.

4. **`makeSampleDotStyleFn` is not used after Phase 64** but is not deleted until Phase 65 cleanup (CONTEXT.md deferred).
