The bug: removing a filter chip (county, ecoregion, or taxon clear) causes a brief flash of ALL specimens before settling to the correct filtered view.

**Root cause analysis (perform these checks to confirm before fixing):**

1. Check for the async race condition in `_runFilterQuery()`. Currently, when `_onFilterChanged` fires:
   - `_filterState` is updated synchronously (triggers Lit re-render)
   - `_runFilterQuery()` is called but is async (DuckDB query)
   - Meanwhile, Lit pushes new `filterState` to bee-map, which triggers `bee-map.updated()` line 170: `regionLayer.changed()`
   - `regionLayer.changed()` forces OpenLayers to repaint ALL layers on the canvas
   - During this repaint, the cluster style function reads `this.visibleEcdysisIds` via closure, which is STALE (old filter result or null)
   - When `_runFilterQuery()` finally resolves, it sets the correct `_visibleEcdysisIds`, triggering another paint

2. Check for concurrent `_runFilterQuery()` calls. Multiple code paths call it: `_onFilterChanged`, `_onRegionClick`, `_onMapClickEmpty`, `_onDataLoaded`, `_onSampleDataLoaded`, `_onPopState` (indirect via `_runFilterQuery`). If two overlap, the first to resolve writes stale results.

3. In bee-filter-controls, check if `_removeCounty` / `_removeEcoregion` / `_clearTaxon` triggers any secondary events (e.g., input blur handler) that could cause a double-fire of `filter-changed`.

**Fix (apply after confirming root cause):**

Add a monotonic generation counter to `_runFilterQuery` in bee-atlas.ts:

```typescript
// Add as private field (non-reactive, not @state):
private _filterQueryGeneration = 0;
```

Update `_runFilterQuery` to guard against stale results:

```typescript
private async _runFilterQuery(): Promise<void> {
  const generation = ++this._filterQueryGeneration;
  const { ecdysis, samples } = await queryVisibleIds(this._filterState);
  // Discard result if a newer query has started since this one began
  if (generation !== this._filterQueryGeneration) return;
  this._visibleEcdysisIds = ecdysis;
  this._visibleSampleIds = samples;
}
```

Additionally, to prevent the OL repaint-before-query-resolves flicker, in `_onFilterChanged` set `_visibleEcdysisIds` and `_visibleSampleIds` to a "pending" sentinel BEFORE starting the async query. The simplest approach: do NOT clear the old IDs synchronously. The generation guard ensures the correct result overwrites. The stale IDs from the previous filter are a better visual state than null (which shows everything).

If the root cause is specifically that `regionLayer.changed()` in bee-map forces a full canvas repaint with stale visibleEcdysisIds, the generation guard alone fixes it — the stale IDs show the OLD (slightly wrong but close) filter, then the correct IDs arrive and trigger a clean repaint.

If the root cause is a double-fire of filter-changed (hypothesis 3), also add a debounce or event.stopPropagation() at the point of redundancy. Document which hypothesis was confirmed in a code comment.
