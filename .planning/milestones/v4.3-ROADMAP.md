# Milestone v4.3: Loading Performance

**Status:** ✅ SHIPPED 2026-05-28
**Phases:** 121–122
**Total Plans:** 5

## Overview

Eliminated the runtime INSERT loop (92K WASM→JS callbacks) by replacing hyparquet+INSERT with a prebuilt SQLite DB (`occurrences.db`) loaded via MemoryVFS, and pre-serializing GeoJSON geometry data at Python export time to reduce the SQL geo query from 570 ms to 80 ms. tablesReady improved 73% (930 ms → 250 ms); loading screen lifted 40% faster (1460 ms → 875 ms). All six PERF requirements satisfied.

## Phases

### Phase 121: Prebuilt SQLite Load

**Goal**: Replace the runtime INSERT loop with a pre-built `occurrences.db` fetched at page load and loaded into MemoryVFS, cutting loading-screen time from ~2.3 s to ~750 ms
**Depends on**: Phase 120
**Requirements**: PERF-01, PERF-02, PERF-03
**Plans**: 3 plans

Plans:

- [x] 121-01-PLAN.md — Pipeline: add `generate_sqlite` step to `data/run.py`; extend `nightly.sh` to upload `occurrences.db` and inject `occurrences_db` key into manifest
- [x] 121-02-PLAN.md — Worker cutover: replace `sqlite-worker.ts` body with MemoryVFS-seeding approach from spike; delete parquet loading, `_insertRows`, `_escapeSqlValue`, `_buildGeoJSON`
- [x] 121-03-PLAN.md — Cleanup and verification: remove hyparquet from worker bundle; benchmark confirm; `npm test` green

**Details:**
Phase 121 shipped `data/sqlite_export.py` — DuckDB sqlite extension exports `occurrences.parquet` to `occurrences.db` with schema derived at runtime (no hardcoded DDL). `nightly.sh` uploads content-hashed and adds `occurrences_db` manifest key. Worker cutover rewrote `sqlite-worker.ts` to 3-step fetch→seed→query, deleting 130 lines of hyparquet+INSERT boot code. Benchmark (Firefox warm): tablesReady 930 ms after Phase 121 alone; main bottleneck identified as SQL geo query (570 ms full table scan with 92K WASM→JS callbacks). hyparquet not fully removed — `bee-map.ts` still imports `parquetReadObjects` for checklist.parquet; full removal deferred to checklist-layer migration phase.

### Phase 122: Worker GeoJSON Aggregation

**Goal:** Eliminate 92K WASM→JS row callbacks by aggregating occurrences into JSON inside SQL (json_group_array), then transferring the result as a zero-copy ArrayBuffer; SQL geo query time drops from ~570 ms to ~50 ms, transfer drops from ~100 ms to near zero
**Requirements**: PERF-GEO-01, PERF-GEO-02, PERF-GEO-03
**Depends on:** Phase 121
**Plans:** 2 plans

Plans:

- [x] 122-01-PLAN.md — sqlite-worker.ts GEO_AGG_SQL + ArrayBuffer transfer; sqlite.ts + features.ts decode + _buildGeoJSONFromRaw; unit tests [PERF-GEO-01, PERF-GEO-02, PERF-GEO-03]
- [x] 122-02-PLAN.md — Regenerate local manifest + occurrences.db; Firefox browser benchmark checkpoint [PERF-GEO-02]

**Details:**
Phase 122 Plan 01 implemented `json_group_array(json_object(...))` — but benchmarked at 1286 ms (2× worse than baseline). Root cause: WASM→JS callback overhead (~6.4 μs × 92,802 rows = ~594 ms) is constant regardless of SQL complexity; SQLite WASM JSON serialization adds additional overhead. A covering index (`idx_occ_geo`) was tried and abandoned — same result because callbacks dominate, not the scan. Fix (Plan 02): pre-serialize geo rows at build time in `sqlite_export.py` using Python `json.dumps`; stored as single TEXT row in `geo_blob` table. Worker queries `SELECT data FROM geo_blob` — 1 callback. Benchmark: SQL geo query 570 ms → 80 ms (86%); tablesReady 930 ms → 250 ms (73%); loading screen 1460 ms → 875 ms (40%). All PERF-GEO targets met.

## Milestone Summary

**Key Decisions:**

- Schema derived from parquet at export time (`CREATE TABLE AS SELECT * FROM read_parquet(...)`) — no hardcoded DDL; future dbt column changes require no edits to `sqlite_export.py`
- MemoryVFS seeding pattern: `mapNameToFile({flags: 0x2, size, data})` before `open_v2` — preloaded DB opens in ~1–3 ms vs ~1229 ms INSERT loop
- `json_group_array` rejected after benchmarking (1286 ms = 2× worse than baseline); root cause is WASM→JS callback overhead, not table scan; `geo_blob` pre-serialization is the correct fix
- Pre-serialized `geo_blob` table: Python `json.dumps` at export time; worker fetches 1 row, 1 callback — 570 ms → 80 ms (86% reduction)

**Issues Resolved:**

- 92K WASM→JS callbacks eliminated as dominant loading bottleneck
- Worker boot time reduced from ~2.3 s baseline (INSERT loop) to ~875 ms total loading screen lifted

**Issues Deferred:**

- hyparquet not fully removed from `package.json` — `bee-map.ts` still imports `parquetReadObjects` for checklist.parquet; full removal requires a separate checklist-layer migration phase
- Phase 121 success criteria (tablesReady ≤ 600 ms, loading screen ≤ 1000 ms) required Phase 122 to be fully met; both targets now satisfied together (250 ms / 875 ms)

**Technical Debt Incurred:**

- None — performance improvements are clean architectural changes
