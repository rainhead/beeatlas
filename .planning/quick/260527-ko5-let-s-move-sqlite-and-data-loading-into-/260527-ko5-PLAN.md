<objective>
Move wa-sqlite, MemoryVFS, parquet parsing, the INSERT pipeline, and all query execution off the main thread into a dedicated module Worker. Capture timing and main-thread heap before and after so we can decide whether the move actually helps.

Purpose: Loading occurrences today blocks the main thread on hyparquet parse + thousands of batched INSERTs through wa-sqlite's Asyncify exec. We want to verify (a) whether the move improves perceived boot time / responsiveness, and (b) whether main-thread memory shrinks meaningfully.

Output: Worker-backed SQLite, façade preserving the existing API surface, and a PROFILE.md with before/after numbers.
</objective>

<task type="auto">
  <name>Task 2: Move SQLite + parquet load into a dedicated Worker behind an RPC façade</name>
  <files>src/sqlite-worker.ts, src/sqlite.ts, vite.config.ts</files>
  <action>
Move all wa-sqlite work off the main thread while preserving the exact public surface of src/sqlite.ts so filter.ts, features.ts, and bee-atlas.ts need no changes.

Architecture:
- New file `src/sqlite-worker.ts` runs as a module Worker. It owns SQLiteESMFactory, the MemoryVFS registration, the open_v2(':memory:') db handle, the serialized exec queue (move `_serializedExec` verbatim), `_insertRows`, `_escapeSqlValue`, and `loadOccurrencesTable`. The worker imports `resolveDataUrl` from `./manifest.ts` (manifest fetch is fine to run inside the worker — it is plain fetch).
- The worker handles three message kinds via a request/response protocol keyed by an incrementing `id`:
  1. `{ id, kind: 'init' }` -> response `{ id, ok: true }` after WASM is instantiated and db opened. Worker auto-runs `loadOccurrencesTable` after init completes and posts `{ kind: 'tables-ready' }` (no id, broadcast) when done. The same broadcast carries the existing `[BENCHMARK]` lines as already-formatted strings inside an `events: string[]` field so the main thread can console.log them — keeping the benchmark output visible in DevTools.
  2. `{ id, kind: 'exec', sql }` -> response `{ id, ok: true, columns: string[], rows: unknown[][] }` (or `{ id, ok: false, error: string, code?: number }`). The worker accumulates rows from the exec callback into a single array; no streaming. All current queries are bounded (PAGE_SIZE=100 or per-page summaries) — bulk row sets only appear in `queryAllFiltered` (CSV) and `loadOccurrenceGeoJSON` (full table for map features). Posting the array once is acceptable and avoids streaming complexity.
  3. (Optional, only if needed for cleanup) `{ id, kind: 'close' }` — skip for now; the page lifecycle handles it.
- Main-thread `src/sqlite.ts` becomes a thin RPC client:
  - Construct the worker with `new Worker(new URL('./sqlite-worker.ts', import.meta.url), { type: 'module' })`.
  - Keep `tablesReady` as a Promise resolved when a `tables-ready` broadcast arrives.
  - `getDB()` returns `{ sqlite3, db }` where `db` is the literal number `0` (the worker keeps the real handle internally; main thread does not need it) and `sqlite3.exec(_db, sql, cb?)` is implemented as: post `{ id, kind: 'exec', sql }`, await the response by id, then synchronously call `cb(row, columns)` for each row in the response. This faithfully reproduces the existing callback contract used by every call site.
  - `loadOccurrencesTable()` becomes essentially `await tablesReady;` — the worker triggers loading itself on init. Keep the exported function so external callers (`bee-atlas.ts` firstUpdated) work unchanged.
  - Print any `events` strings forwarded from the worker via console.log so existing `[BENCHMARK]` lines still show up in DevTools.
  - Add one more main-thread benchmark on init: log `[BENCHMARK] worker boot RPC roundtrip: R ms` measured from worker construction to receipt of the `tables-ready` broadcast.
- Vite: confirm the worker is bundled. Vite handles `new Worker(new URL(...), { type: 'module' })` natively, but wa-sqlite's `.wasm` and `.mjs` must still resolve from inside the worker. The existing `optimizeDeps: { exclude: ['wa-sqlite'] }` should remain. If the worker build fails on the wa-sqlite import, add `worker: { format: 'es' }` to vite.config.ts (do not switch to classic workers).
- Serialization detail: the row callback in current call sites is synchronous and pushes to a local array — that is safe to call in a loop on the main thread after the response arrives. Confirm no call site relies on rows arriving incrementally (none do — they all push then return).
- Keep the existing `wa-sqlite/src/examples/MemoryVFS.js` import inside the worker; do not switch VFS.
- Do NOT introduce IndexedDB persistence, OPFS, SharedArrayBuffer, or any other "while we're in here" change. Only the thread move.
- Update test mocks if the surface shape changes: in `src/tests/filter.test.ts` and `src/tests/bee-atlas.test.ts`, the existing mocks return `{ sqlite3: { exec: vi.fn(...) }, db: 0 }` — that shape is preserved by the façade, so the mocks should keep working unmodified. Verify by running `npm test`. If a test breaks because the façade does something at module-load time (e.g. constructs the worker eagerly), defer worker construction until first call to `getDB()` so tests that mock `./sqlite.ts` never instantiate the worker.

Order of operations inside the file:
1. Create `src/sqlite-worker.ts` first (full implementation including moved helpers).
2. Replace `src/sqlite.ts` body with the façade. Keep the exported names and types identical.
3. Adjust `vite.config.ts` only if the dev server or `npm run build` fails on the worker.
